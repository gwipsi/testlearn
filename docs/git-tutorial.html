⚡ Schnelleinstieg: Die 7 wichtigsten Git-Befehle

Du hast 2 Minuten? Hier sind die Befehle, die du täglich brauchst:

  1. git clone <url> – Repository herunterladen
  2. git checkout -b feature/name – Neuen Branch erstellen
  3. git add . – Änderungen vorbereiten
  4. git commit -m "Beschreibung" – Snapshot speichern
  5. git push origin feature/name – Branch hochladen
  6. git pull origin main – Neuesten Stand holen
  7. git merge feature/name – Branch zusammenführen

💡 Tipp: Alle 7 Befehle in dieser Reihenfolge decken 95% deiner täglichen Arbeit ab.

Was ist Git?

Git ist ein Versionskontrollsystem (Version Control System, kurz VCS). Es speichert nicht nur die aktuelle Version deiner Dateien, sondern die gesamte Geschichte aller Änderungen – wer wann was geändert hat, und warum.

Kernidee in einem Satz: Git ist wie ein unbegrenztes „Rückgängig" für dein gesamtes Projekt – über Monate, Teams und Computer hinweg.

Git wurde 2005 von Linus Torvalds (dem Schöpfer von Linux) entwickelt, weil kein bestehendes Tool schnell genug für die Linux-Kernel-Entwicklung war. Heute ist Git der weltweite Standard in der Softwareentwicklung.

Was Git löst:

  • „Welche Version hatte die Funktion, die noch funktioniert hat?"
  • „Kollege und ich haben dieselbe Datei gleichzeitig bearbeitet – was jetzt?"
  • „Ich möchte ein neues Feature ausprobieren, ohne die funktionierende Version zu zerstören."
  • „Wer hat diese Zeile wann und warum geändert?"
🔧 Im IT-Systemhaus Du hast für Kunde Müller GmbH ein PowerShell-Monitoring-Skript entwickelt. Drei Monate später ruft der Kunde an: „Seit dem letzten Windows-Update liefert das Skript falsche Werte." Mit Git siehst du in Sekunden, welche Änderung wann gemacht wurde und kannst den alten Stand wiederherstellen – anstatt in Dateien wie Get-Info_v3_FINAL_neu2.ps1 zu suchen.

Die technische Basis – wie Git wirklich funktioniert

Viele denken, Git speichere einfach Dateiversionen wie ein Papierkorb. In Wirklichkeit ist Git ein inhaltsadressierbares Dateisystem – eine elegante Objektdatenbank. Das Verständnis davon erklärt, warum Git so mächtig ist.

Der .git-Ordner – das Herz des Repositories

In jedem Git-Projekt gibt es einen versteckten .git-Ordner. Darin liegt alles:

.git/ ├── objects/ # Die eigentliche Datenbank – alle Objekte │ ├── 1a/ │ │ └── b4c5d9... # Ein Objekt (Blob: Inhalt von index.html) │ ├── 3f/ │ │ └── e7a2b1... # Ein Objekt (Tree: Verzeichnis /docs) │ └── c8/ │ └── 2eab66... # Ein Objekt (Commit: "Add stars") ├── refs/ │ └── heads/ │ ├── main # Enthält: c82eab6... (nur ein SHA-Zeiger!) │ └── feature # Enthält: 5b96f9f... └── HEAD # Enthält: ref: refs/heads/main
Wichtige Erkenntnis: Die Dateien in deinem Projektordner (z.B. docs/index.html) sind nur eine Ansicht des aktuellen Commits. Die echten Daten liegen komprimiert im .git/objects/-Ordner.

Die 4 Git-Objekttypen

Blob

Der rohe Dateiinhalt – z.B. der gesamte Text einer JavaScript-Datei. Kein Dateiname, kein Pfad – nur reiner Inhalt.

Tree

Eine Verzeichnisliste: zeigt auf Blobs (Dateien) und andere Trees (Unterordner). Entspricht einem Ordner in deinem Projekt.

Commit

Ein Snapshot: zeigt auf einen Tree (Zustand des Projekts), den Parent-Commit, Autor, Zeitstempel und Beschreibung.

Tag

Ein benannter Zeiger auf einen bestimmten Commit – z.B. v1.0.0 für eine Release-Version.

SHA-1-Hashing – der Fingerabdruck

Jedes Objekt wird durch den SHA-1-Hash seines Inhalts adressiert (z.B. c82eab66b20d06953e...). Ändert sich auch nur ein einziges Zeichen, entsteht ein komplett neuer Hash. Das garantiert:

  • 🔒 Integrität – Du kannst sofort erkennen, ob ein Objekt verändert wurde
  • ⚡ Effizienz – Identische Dateien werden nur einmal gespeichert
  • ⏮️ Unveränderlichkeit – Alles bleibt erhalten, nichts wird überschrieben

Der Commit-Graph (DAG)

Commits bilden einen gerichteten azyklischen Graphen (Directed Acyclic Graph). Jeder Commit zeigt auf seinen Vorgänger. Branches sind dabei nur leichtgewichtige Zeiger – eine simple Textdatei mit einem SHA-Hash:

A ← B ← C ← F (main) \ / D ← E (feature/login) A, B, C... = einzelne Commits (unveränderliche Snapshots) F = Merge-Commit (hat zwei Parents: C und E) main = Zeiger auf F (nur ein SHA-Hash, kein echter "Zweig") feature = Zeiger auf E
Das erklärt, warum man immer zurück kann: Da Commits nie gelöscht werden (nur Zeiger bewegen sich), ist jeder frühere Zustand des Projekts dauerhaft im .git/objects/-Ordner gespeichert und erreichbar.

Kernbegriffe von A–Z

Repository (Repo)

Der Projektordner inkl. dem versteckten .git-Unterordner. Der .git-Ordner enthält das gesamte Versionsarchiv.

Ein Projektstammbuch: vorne die aktuelle Version, hinten alle früheren Revisionen seit Beginn.

# Neues Repository erstellen git init # Vorhandenes Repository klonen git clone https://github.com/user/repo.git

Clone

Eine vollständige lokale Kopie eines Remote-Repositories – inklusive der gesamten Geschichte aller Commits.

Du bestellst das komplette Buch inklusive aller Entwürfe auf deinen PC. Du hast danach alles lokal.

git clone https://github.com/gwipsi/testlearn.git # → Erstellt Ordner "testlearn/" mit allen Daten

Commit

Ein gespeicherter Snapshot des Projekts mit Beschreibung, Autor und Zeitstempel. Das Fundament von Git.

Quicksave in einem Videospiel – du kannst jederzeit zu genau diesem Punkt zurück.

# 1. Dateien für den Commit vorbereiten (Staging) git add scripts/Get-SystemInfo.ps1 # 2. Snapshot mit Beschreibung speichern git commit -m "Add disk space check to monitoring script"

🔧 Kunde Müller bekommt eine neue Funktion – jetzt committen, damit die Änderung sauber dokumentiert ist, bevor es zum nächsten Feature geht.

Branch

Ein unabhängiger Entwicklungsstrang – eine eigene „Zeitleiste" für ein Feature oder einen Bugfix. Technisch nur ein Zeiger auf einen Commit.

Eine Kopie des Dokuments zum Experimentieren, ohne das Original zu berühren.

# Neuen Branch erstellen und wechseln git checkout -b feature/cpu-temperatur-wagner # Zurück zu main (Basisskript für alle anderen) git checkout main # Alle Branches anzeigen git branch

🔧 Dein Standard-Skript läuft bei 5 Kunden. Kunde Wagner will CPU-Temperatur – aber nur er hat die passenden Sensoren. Branch für Wagner, der Rest bleibt unberührt.

Push

Lokale Commits auf den Remote-Server (z.B. GitHub) hochladen, damit andere sie sehen können.

Deine gespeicherten Änderungen von deinem PC in die Cloud synchronisieren.

# Branch auf Remote hochladen git push origin feature/neue-funktion # Beim ersten Mal mit -u (setzt Upstream) git push -u origin feature/neue-funktion

Pull

Neueste Änderungen vom Remote-Server holen und in den lokalen Branch einbauen. Kurzform für git fetch + git merge.

Updates aus der Cloud auf deinen PC synchronisieren.

# Neuesten Stand von main holen git pull origin main # Oder kurz, wenn Upstream gesetzt ist git pull

Merge

Zwei Branches zusammenführen. Git kombiniert automatisch die Änderungen – bei Konflikten muss man manuell helfen.

Die Änderungen deiner Testkopie werden ins Original übernommen.

# Zuerst zum Ziel-Branch wechseln git checkout main # Dann den Feature-Branch einbauen git merge feature/neue-funktion

Pull Request (PR)

Ein Antrag auf GitHub, eine Branch in eine andere zu mergen. Ermöglicht Code-Review, Diskussion und automatische Tests bevor etwas in main landet.

Du reichst deinen Änderungsvorschlag zur Prüfung ein. Kollegen können kommentieren und freigeben.

# Ablauf: Branch pushen → PR auf GitHub erstellen git push origin feature/neue-funktion # → Dann auf github.com den PR eröffnen

Fork

Eine eigene Kopie eines fremden Repositories auf GitHub – auf Server-Ebene, nicht nur lokal. Typisch bei Open-Source-Beiträgen.

Du legst eine Kopie des ganzen Projekts in deinem eigenen GitHub-Account an und arbeitest dort.

Ablauf: Fork auf GitHub → lokaler Clone → Änderungen → PR an das Original-Repo

Revert

Erzeugt einen neuen Commit, der einen bestimmten früheren Commit exakt rückgängig macht. Die sicherste „Undo"-Methode für bereits gepushte Commits.

Du machst eine Änderung rückgängig, aber das Protokoll zeigt trotzdem, dass es sie gab.

# Einen bestimmten Commit rückgängig machen git revert c82eab6 # Ohne automatischen Commit (manuell bestätigen) git revert --no-edit c82eab6

Staging Area

Eine Zwischenebene zwischen Working Directory und Commit. Du entscheidest bewusst, welche Änderungen in den nächsten Commit kommen.

Du legst Sachen in einen Karton, bevor du ihn abschickst – du kannst noch wählen, was reinkommt.

# Nur diese Datei für den Commit vorbereiten git add docs/styles.css # Alle geänderten Dateien vorbereiten git add . # Zeigen, was in der Staging Area liegt git status

HEAD

Ein spezieller Zeiger, der angibt, wo du gerade „bist" – welcher Branch oder Commit aktiv ist. .git/HEAD ist schlicht eine Textdatei.

Die „Sie sind hier"-Markierung auf einer Karte.

# HEAD zeigt auf den aktuellen Branch cat .git/HEAD # Ausgabe: ref: refs/heads/main # Commits relativ zu HEAD ansehen git log HEAD~3..HEAD

Lokal vs. Remote – die zwei Welten

Git ist ein verteiltes Versionskontrollsystem: Jeder Entwickler hat eine vollständige Kopie des Repositories auf seinem PC. Es gibt keinen zentralen Server, der unbedingt gebraucht wird – GitHub ist bequem, aber kein Muss.

Dein PC (lokal) GitHub (remote) +-------------------------------+ +---------------------------+ | | | | | Working Directory | | origin/main | | (deine sichtbaren Dateien) | | origin/feature/... | | | | | | | git add | push | (alle gepushten Commits | | | | ====> | und Branches) | | Staging Area | | | | | | <==== | | | git commit | pull | | | | | fetch | | | Local Repository (.git/) | | | | (vollständige History!) | | | +-------------------------------+ +---------------------------+

Lokal – was du ohne Internet tun kannst

  • Commits erstellen (git commit)
  • Branches anlegen und wechseln
  • Die gesamte Historie einsehen
  • Dateien vergleichen (git diff)
  • Commits rückgängig machen (git revert)

Remote – was eine Verbindung braucht

  • Commits hochladen (git push)
  • Commits herunterladen (git pull / fetch)
  • Pull Requests erstellen
  • Branches mit Teamkollegen teilen
  • CI/CD-Pipelines auslösen
Praxistipp: Committe regelmäßig lokal, auch wenn du noch nicht pushen willst. Lokale Commits kosten nichts und sichern deinen Fortschritt – gepusht wird, wenn du bereit bist.

Im Team arbeiten – der Git-Workflow

Der bekannteste Workflow für Teams ist der Feature-Branch-Workflow: Jedes neue Feature oder jeder Bugfix bekommt einen eigenen Branch. Nur fertige, geprüfte Änderungen kommen in main.

Der Standard-Ablauf

  1. 1
    Neuesten Stand holen
    git pull origin main – bevor du anfängst, immer synchronisieren.
  2. 2
    Eigenen Branch erstellen
    git checkout -b feature/login-button – du arbeitest isoliert, ohne main zu berühren.
  3. 3
    Coden und committen
    Regelmäßig kleine, sinnvolle Commits machen. Nicht alles auf einmal.
  4. 4
    Branch pushen
    git push -u origin feature/login-button – der Branch ist jetzt auf GitHub sichtbar.
  5. 5
    Pull Request erstellen
    Auf GitHub: PR von deinem Branch in main. Kollegen können reviewen und kommentieren.
  6. 6
    Review & Merge
    Nach Freigabe wird der Branch in main gemergt. CI-Tests laufen automatisch.
  7. 7
    Branch löschen
    Gemergter Branch kann gelöscht werden – er wird nicht mehr gebraucht.

Was tun bei Merge-Konflikten?

Wenn zwei Personen dieselbe Zeile einer Datei geändert haben, entsteht ein Merge-Konflikt. Git markiert die konfliktierenden Stellen:

<<<<<<< HEAD (deine Version) color: #8B0000; ======= color: #0078D4; >>>>>>> feature/neue-farbe (andere Version)

Du entscheidest manuell, welche Version (oder eine Kombination) korrekt ist, entfernst die Markierungen, und erstellst dann einen neuen Commit.

Konflikte vermeiden: Häufig pullen (git pull) und in kleinen, überschaubaren Einheiten arbeiten. Je länger ein Branch lebt, ohne mit main synchronisiert zu werden, desto größer werden Konflikte.
🔧 Im IT-Systemhaus: geteiltes Inventarisierungs-Tool Du und Kollege Max entwickelt gemeinsam ein Tool zur Inventarisierung von Kundenhardware. Max schreibt die PowerShell-Logik (feature/ps-abfrage), du baust das HTML-Dashboard (feature/dashboard). Beide arbeiten parallel ohne Konflikte. Wenn Max fertig ist, macht er einen PR – du reviewst, ob seine Datenstruktur zu deinem Dashboard passt. Erst dann wird gemergt.

Mehrere Projekte – mehrere Repositories

In einem Team arbeiten oft mehrere Personen an verschiedenen Projekten gleichzeitig. Jedes Projekt hat sein eigenes Repository.

Typische Struktur lokal

~/projekte/ ├── kunde-mueller-monitoring/ # Monitoring-Skripte Müller GmbH │ └── .git/ ├── kunde-wagner-inventar/ # Inventarisierungstool Wagner KG │ └── .git/ ├── intern-helpdesk-tool/ # Internes AD-Abfrage-Tool │ └── .git/ └── vorlagen-bibliothek/ # Wiederverwendbare PS-Funktionen └── .git/

Jeder Ordner ist ein eigenständiges Repository. Git-Befehle beziehen sich immer auf das Repo im aktuellen Verzeichnis.

Wer arbeitet woran?

👩‍💻 Alice arbeitet im frontend-Repo an einer neuen Seite.
👨‍💻 Bob arbeitet im backend-api-Repo an einem neuen Endpoint.
Beide können pushen, ohne sich gegenseitig zu stören.

Wenn Alices Frontend die neue API von Bob braucht: Sie stimmen sich über den Branch-Stand ab und Alice pullt zur richtigen Zeit.

GitHub-Organisation: Wenn ein ganzes Team viele Repos verwaltet, legt man auf GitHub eine Organisation an (z.B. github.com/mein-unternehmen). Alle Repos der Firma liegen dort gesammelt, mit zentraler Rechteverwaltung.

Windows-Werkzeuge – womit arbeitest du?

Empfohlen

VS Code + GitLens

Visual Studio Code hat eine eingebaute Git-Unterstützung (Source Control Tab, linke Leiste). Mit der Extension GitLens sieht man direkt im Code, wer wann was geändert hat (Git Blame). Das integrierte Terminal erlaubt git-Befehle direkt nebenher.

# GitLens im VS Code Extension-Tab suchen: GitLens — Git supercharged
Für Einsteiger

GitHub Desktop

Eine grafische Oberfläche speziell für Git – kein Terminal nötig. Zeigt Änderungen visuell (grün = hinzugefügt, rot = entfernt), macht Commits, Pushs und PRs per Klick. Ideal zum Einstieg.

Download: desktop.github.com

Profis

Terminal / PowerShell

Git für Windows installieren (git-scm.com), dann sind alle git-Befehle in PowerShell und CMD verfügbar. Am flexibelsten, läuft auf jedem System identisch. VS Code hat ein eingebautes Terminal, das direkt dafür genutzt werden kann.

# Git-Version prüfen git --version
Nützlich

Windows Terminal

Moderner Ersatz für die alte Windows-Konsole. Unterstützt mehrere Tabs, PowerShell, CMD und WSL in einem Fenster. Empfohlen als Standard-Terminal unter Windows 11.

Download: Microsoft Store → „Windows Terminal"

Empfohlenes Setup für Windows-Entwickler: Git für Windows installieren → VS Code installieren → GitLens-Extension aktivieren → fertig. Das Terminal in VS Code ist direkt nutzbar.

Mit KI-Agenten arbeiten – wie Claude Code Git nutzt

Wenn du mit einem KI-Agenten wie Claude Code arbeitest, interagiert dieser direkt mit Git in deinem Repository – ähnlich wie ein Kollege, der per Fernzugriff am selben Projekt arbeitet.

Was passiert technisch im Hintergrund?

GitHub-Repository (origin) | | git clone (beim Session-Start) ↓ Remote Sandbox (Cloud-Container) ├── Working Directory (geklontes Repo) ├── .git/ (vollständige History) └── Claude Code (hat Zugriff auf alle git-Befehle) | | git add, git commit, git push ↓ GitHub-Repository (origin) ← Änderungen landen hier

Was der Agent tun kann

  • Branches erstellen und wechseln
  • Dateien lesen, bearbeiten, erstellen
  • Commits mit sinnvollen Nachrichten erstellen
  • Branches pushen
  • Mergen und reverten
  • Git-Historie analysieren

Was der Agent nicht tut (ohne Erlaubnis)

  • Force-Push auf main (--force)
  • Destructive operations ohne Nachfragen
  • Direkt auf main committen (normalerweise)
  • PRs ohne explizite Anfrage erstellen
Wichtig zu wissen: Git-Hooks im Agenten Lokale Git-Hooks (z.B. ein Pre-Commit-Hook, der Timestamps automatisch aktualisiert) laufen möglicherweise nicht in der Remote-Sandbox – der Hook liegt zwar in .git/hooks/, aber Hooks werden bei einem frischen Clone oft nicht automatisch ausgeführt. Das erklärt, warum manche automatischen Aktionen (wie Timestamp-Updates) beim Agenten-Commit ausbleiben können.

Wie du sinnvoll mit einem Agenten zusammenarbeitest

  1. ✓
    Klare Aufgaben beschreiben – „Ändere die Farbe der Überschriften auf Dunkelrot" ist besser als „mach die Seite schöner".
  2. ✓
    Branch-Strategie besprechen – soll der Agent direkt auf main oder auf einem Feature-Branch arbeiten?
  3. ✓
    Vor großen Änderungen nachfragen – ein guter Agent fragt bei riskanten Aktionen nach, bevor er sie ausführt.
  4. ✓
    Geschichte im Blick behalten – git log zeigt dir genau, was der Agent committet hat.

Best Practices – sauber arbeiten mit Git

Gute Commit-Nachrichten

Eine gute Nachricht beschreibt das Warum, nicht nur das Was. Im Imperativ schreiben.

# ❌ Schlecht git commit -m "fix stuff" git commit -m "changes" # ✅ Gut git commit -m "Fix crash when profile photo is missing" git commit -m "Add dark-mode support to dashboard"

Wann committen?

Ein Commit = eine logische Einheit. Nicht zu groß, nicht zu klein. Lieber öfter als zu selten.

  • ✅ Funktion fertig und getestet
  • ✅ Bugfix abgeschlossen
  • ✅ Bevor du ein Experiment startest
  • ❌ Mitten in einer halbfertigen Funktion
  • ❌ Dateien, die nicht dazugehören

Branch-Benennung

Konsistente Namen machen die Übersicht einfacher – besonders im Team.

feature/user-authentication feature/dark-mode fix/login-crash fix/typo-in-header release/v2.0.0 hotfix/critical-security-patch

main immer stabil halten

main sollte immer deploybar sein. Nie direkt auf main entwickeln.

  • ✅ Feature-Branches für jede Änderung
  • ✅ PR-Review vor dem Merge
  • ✅ Automatische Tests (CI) bei jedem PR
  • ❌ Direkte Commits auf main (außer Notfälle)

.gitignore richtig einsetzen

Bestimmte Dateien sollen nie in Git landen: Passwörter, Build-Artefakte, lokale Einstellungen.

# Typische .gitignore-Einträge: node_modules/ .env *.log dist/ .DS_Store Logs/

Regelmäßig pullen

Je länger dein Branch nicht mit main synchronisiert wurde, desto größer werden Merge-Konflikte.

# Am Anfang jedes Arbeitstages: git pull origin main # Oder den eigenen Branch aktualisieren: git fetch origin git rebase origin/main
Das wichtigste Prinzip: Git ist dein Sicherheitsnetz – nicht dein Feind. Je öfter du commitest und pushst, desto weniger kann schiefgehen. Kein Commit ist in Stein gemeißelt: mit git revert oder git reset lässt sich fast jede Aktion rückgängig machen.

Praxis-Durchlauf: Ein Kundenprojekt von A bis Z

Szenario: Kunde Bauer GmbH braucht ein PowerShell-Skript, das täglich Festplattenplatz prüft und eine Log-Datei schreibt. Das Projekt soll sauber mit Git verwaltet werden – auch wenn später Änderungen kommen.

📁

Schritt 1: Repository anlegen

Neues Projekt, neues Repo. Einmalig am Anfang:

mkdir kunde-bauer-diskcheck cd kunde-bauer-diskcheck git init # Auf GitHub: neues Repo erstellen, dann verknüpfen: git remote add origin https://github.com/mein-systemhaus/kunde-bauer-diskcheck.git
✏️

Schritt 2: Erstes Skript, erster Commit

Die Grundstruktur steht. Noch nicht fertig, aber ein sinnvoller Zwischenstand:

# .gitignore anlegen – Logs sollen nie ins Repo echo "Logs/" > .gitignore # Erste Version des Skripts committen git add Get-DiskCheck.ps1 .gitignore git commit -m "Add initial disk check script with basic log output" git push -u origin main
🔧

Schritt 3: Neues Feature – E-Mail-Benachrichtigung

Kunde will bei unter 10 % freiem Speicher eine E-Mail bekommen. Neues Feature = neuer Branch:

git checkout -b feature/email-benachrichtigung # … Funktion Send-Benachrichtigung entwickeln und testen … git add Get-DiskCheck.ps1 git commit -m "Add email alert when disk space below 10%" git push -u origin feature/email-benachrichtigung

Dann auf GitHub einen Pull Request erstellen. Kollege Max reviewt den Code, bevor er produktiv geht.

🔀

Schritt 4: Review und Merge

Max hat die E-Mail-Funktion freigegeben. PR in main mergen, Branch löschen:

git checkout main git merge feature/email-benachrichtigung git push origin main # Feature-Branch aufräumen git branch -d feature/email-benachrichtigung git push origin --delete feature/email-benachrichtigung
🚨

Schritt 5: Hotfix – Skript crasht beim Kunden

Zwei Wochen später: Anruf vom Kunden. Das Skript bricht mit einem Fehler ab. Hotfix direkt von main:

git checkout main git pull origin main git checkout -b hotfix/null-check-laufwerk # Schnell den Fehler beheben git add Get-DiskCheck.ps1 git commit -m "Fix null reference when network drive is unreachable" git push -u origin hotfix/null-check-laufwerk # PR → Merge → fertig. Kein Feature beeinflusst.
🔁

Schritt 6: Zweiter Kunde will dasselbe Skript

Kunde Vogel GmbH braucht dasselbe Tool, aber mit anderen Schwellwerten und einem eigenen Log-Pfad. Optionen:

  • Neues Repo klonen: git clone … kunde-vogel-diskcheck – sauberste Lösung für Kundenprojekte mit eigener Entwicklung
  • Konfiguration auslagern: Werte in eine config.json auslagern, die nicht im Repo liegt (.gitignore) – ein Repo für beide Kunden
  • Branch pro Kunde: Nur sinnvoll bei sehr kleinen Unterschieden und wenn beide Kunden aktiv weiterentwickelt werden
📋

Schritt 7: Dokumentieren im Commit-Log

Nach sechs Monaten schaut ein neuer Kollege ins Repo. Das Commit-Log ist seine Dokumentation:

git log --oneline # Ausgabe: a3f9c12 Fix null reference when network drive is unreachable 8b2e4a1 Add email alert when disk space below 10% 3c7d9f0 Add initial disk check script with basic log output

Drei Zeilen, die die gesamte Geschichte des Projekts erklären. Kein Word-Dokument, kein „v2_FINAL" im Dateinamen.

Das Muster für jedes Projekt im Systemhaus: Neues Projekt → git init + Remote anlegen → kleine Commits von Anfang an → Features in Branches → Kollege reviewed → Merge → Branch löschen. Egal ob das Projekt 2 Stunden oder 2 Jahre dauert.
