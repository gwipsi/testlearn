🎯 Kernkonzepte: Die 7 Prinzipien von Git

Du arbeitest mit KI-Agenten und willst Git verstehen? Hier sind die 7 Konzepte, die dir alles Weitere erklären:

  1. Klone ein Repository – Die komplette Historie laden, damit du nicht von null anfängst
  2. Erstelle einen eigenen Branch – Isolierte Arbeit für jedes Feature, ohne main zu berühren
  3. Verstehe die Staging Area – Bewusste Auswahl, welche Änderungen in den nächsten Snapshot kommen
  4. Committe logische Einheiten – Kleine, aussagekräftige Snapshots, nicht alles auf einmal
  5. Synchronisiere regelmäßig – Hol neueste Änderungen vom Team, bevor Konflikte wachsen
  6. Nutze Pull Requests für Review – Änderungen müssen geprüft werden, bevor sie in main landen
  7. Verstehe Merge-Strategien – Die Art, wie Features zusammengefügt werden, prägt die History

💡 Hinweis: Diese 7 Konzepte sind das Fundament – ohne sie verstehst du nicht, was passiert, wenn du (oder dein Agent) mit Git arbeitet. Alle Details findest du in den Abschnitten unten.

Grundlagen

Die Basis, die jeder kennen muss: Was ist Git, die wichtigsten Begriffe, und das Zusammenspiel von deinem PC mit der Cloud.

1

Was ist Git?

Git ist ein Versionskontrollsystem (Version Control System, kurz VCS). Es speichert nicht nur die aktuelle Version deiner Dateien, sondern die gesamte Geschichte aller Änderungen – wer wann was geändert hat, und warum.

In einem Satz: Git ist wie ein unbegrenztes „Rückgängig" für dein gesamtes Projekt – speichere, experimentiere, und kehre jederzeit zu einem früheren Stand zurück, über Monate, Teams und Computer hinweg.

Was Git im Alltag löst:

  • „Welche Version hatte die Funktion, die noch funktioniert hat?"
  • „Kollege und ich haben dieselbe Datei gleichzeitig bearbeitet – was jetzt?"
  • „Ich möchte etwas Neues ausprobieren, ohne die funktionierende Version zu zerstören."
  • „Wer hat diese Zeile wann und warum geändert?"
🔧 Im IT-Systemhaus Du hast für Kunde Müller GmbH ein PowerShell-Monitoring-Skript entwickelt. Drei Monate später ruft der Kunde an: „Seit dem letzten Windows-Update liefert das Skript falsche Werte." Mit Git siehst du in Sekunden, welche Änderung wann gemacht wurde und kannst den alten Stand wiederherstellen – anstatt in Dateien wie Get-Info_v3_FINAL_neu2.ps1 zu suchen.

Die wichtigsten Begriffe

Diese acht Begriffe bilden das Vokabular, mit dem du 95% deiner Arbeit erledigst. Weitere (etwas speziellere) Begriffe findest du später in Teil 4.

Repository (Repo)

Der Projektordner mit seinem versteckten .git-Verzeichnis – einem digitalen Archiv der gesamten Projektgeschichte.

Ein Projekt-Tagebuch, das jeden Stand und jede Änderung seit Anfang dokumentiert. Die aktuelle Version ist nur die „aktuelle Seite".

Was ein Agent verstehen muss: Ein Repository ist die Verwaltungseinheit für alle Änderungen. Er kann es klonen, darin arbeiten, und Änderungen sichern – alles in einem strukturierten System.

Clone

Eine komplette, lokale Kopie eines Repositories – mit der vollständigen Geschichte aller je gemachten Commits.

Du bestellst das komplette Buch inkl. aller Entwürfe. Du hast danach alles lokal verfügbar – brauchst kein Netzwerk mehr für die Geschichte.

Was ein Agent verstehen muss: Cloning ist der Start jeder Arbeit. Nach dem Clone hat der Agent alles, was er braucht – die komplette Historie, alle Branches, die komplette Architektur.

Commit

Ein unveränderlicher Snapshot des Projekts mit Beschreibung (Message), Autor und Zeitstempel. Commits sind das Fundament von Git – Bausteine der Geschichte.

Quicksave in einem Videospiel – du speicherst den Punkt, zu dem du jederzeit zurückkehren kannst. Jeder Save ist permanent dokumentiert.

Was ein Agent verstehen muss: Ein guter Commit ist eine logische, abgeschlossene Einheit. Die Message ist für Menschen – nicht nur eine Liste von Dateien.

Branch

Ein unabhängiger Entwicklungsstrang – eine eigene Zeitleiste für ein Feature, Bugfix oder Experiment. Technisch: ein Zeiger auf einen Commit, mehr nicht.

Eine Kopie des Dokuments, auf der du arbeitest, ohne das Original zu berühren. Alle Kopien können parallel existieren, ohne sich zu beeinflussen.

Was ein Agent verstehen muss: Branches sind die Isolationseinheit. Jede Aufgabe (Feature, Fix) bekommt einen Branchfür sich allein – damit arbeitet der Agent sauber und unabhängig.

Push

Lokale Commits auf den Remote-Server (z.B. GitHub) hochladen, damit das Team und die Welt sie sehen können.

Deine fertigen, lokal gespeicherten Änderungen in die Cloud synchronisieren – damit andere Teams-Mitglieder darauf aufbauen können.

Was ein Agent verstehen muss: Push ist die Verbindung zwischen lokal und Remote. Nur gepushte Commits sind für andere erreichbar.

Pull

Neueste Änderungen vom Remote-Server holen und in den lokalen Branch integrieren. Technisch: fetch (herunterladen) + merge (einbauen) in einem Schritt.

Updates aus der Cloud auf deinen PC synchronisieren, damit du auf dem neuesten Stand arbeitest.

Was ein Agent verstehen muss: Pull ist die Synchronisation. Der Agent holt regelmäßig Änderungen, damit seine Arbeit nicht veraltet und Konflikte klein bleiben.

Merge

Zwei Branches zusammenführen – Git kombiniert die Änderungen automatisch. Bei Konflikten (wenn beide Seiten dieselbe Zeile änderten) müssen die Unterschiede manuell entschieden werden.

Änderungen aus deiner Testkopie werden ins Original übernommen. Wenn beide Kopien an derselben Stelle geändert wurden, muss man klären, welche Version zählt.

Was ein Agent verstehen muss: Merge ist die Zusammenbringung. Die Strategie (Merge Commit, Squash, Rebase) bestimmt, wie die Geschichte danach aussieht.

Pull Request (PR)

Ein formeller Antrag auf GitHub, einen Branch in einen anderen (meist main) zu mergen. Ermöglicht Code-Review, Diskussion und automatische Tests bevor etwas in production geht.

Du reichst deinen Änderungsvorschlag zur Prüfung ein. Kollegen (oder ein Agent) können kommentieren, Änderungen anfordern und am Ende freigeben.

Was ein Agent verstehen muss: Ein PR ist nicht nur das Mergen – es ist der Diskurs. Ein guter Agent erklärt seine Änderungen in der PR-Beschreibung, damit Menschen es verstehen und prüfen können.

3

Lokal vs. Remote – die zwei Welten

Git ist ein verteiltes Versionskontrollsystem: Jeder Entwickler hat eine vollständige Kopie des Repositories auf seinem PC. Es gibt keinen zentralen Server, der unbedingt gebraucht wird – GitHub ist bequem, aber kein Muss.

Dein PC (lokal) GitHub (remote) +-------------------------------+ +---------------------------+ | | | | | Working Directory | | origin/main | | (deine sichtbaren Dateien) | | origin/feature/... | | | | | | | git add | push | (alle gepushten Commits | | | | ====> | und Branches) | | Staging Area | | | | | | <==== | | | git commit | pull | | | | | fetch | | | Local Repository (.git/) | | | | (vollständige History!) | | | +-------------------------------+ +---------------------------+

Lokal – was du ohne Netzwerk tun kannst

  • Commits erstellen und die Geschichte verfassen
  • Branches anlegen, wechseln und mergen
  • Die gesamte Projekthistorie einsehen
  • Dateien vergleichen und Unterschiede analysieren
  • Commits rückgängig machen oder ungeschehen machen
  • Experimentieren, ohne jemanden zu beeinflussen

Remote – was Netzwerk braucht

  • Commits hochladen (Team sieht deine Arbeit)
  • Commits herunterladen (dein Stand wird aktuell)
  • Pull Requests erstellen und diskutieren
  • Branches mit dem Team teilen und zusammenbringen
  • CI/CD-Pipelines auslösen (automatische Tests)
  • Mehrsprachige, verteilt arbeitende Teams koordinieren
Praxistipp: Committe regelmäßig lokal, auch wenn du noch nicht pushen willst. Lokale Commits kosten nichts und sichern deinen Fortschritt – gepusht wird, wenn du bereit bist.

Teil 2 · Der Alltag

So arbeitest du täglich

Der praktische Kern: der Workflow von Branch bis Merge, wie Änderungen sauber in main kommen, und wie du dabei ordentlich bleibst.

4

Der tägliche Workflow – allein und im Team

Der bekannteste Workflow ist der Feature-Branch-Workflow: Jedes neue Feature oder jeder Bugfix bekommt seinen eigenen Branch. Nur fertige, geprüfte Änderungen kommen in main. Das funktioniert allein genauso wie im Team.

Der Standard-Ablauf

  1. 1
    Synchronisiere mit dem Team
    Bevor du anfängst: hole den neuesten Stand von main. Das verhindert, dass deine Arbeit veraltet ist, bevor du sie anfängst.
  2. 2
    Erstelle einen Feature-Branch
    Ein neuer Branch für deine Aufgabe (z.B. feature/login-button). Du arbeitest isoliert, ohne main zu berühren.
  3. 3
    Implementiere und committen
    Schreibe Code, teste lokal, erstelle regelmäßig kleine, aussagekräftige Commits. Ein Commit = eine logische Einheit, nicht 10 verschiedene Dinge auf einmal.
  4. 4
    Lade deinen Branch hoch
    Push den Branch auf GitHub (oder dein Git-System). Jetzt ist deine Arbeit für andere sichtbar und gesichert.
  5. 5
    Öffne einen Pull Request
    Ein formales „Hey, ich bin fertig – bitte schauen und freigeben." Auf GitHub zeigt der PR genau, welche Zeilen du geändert hast.
  6. 6
    Review und Merge
    Kollegen (oder ein Agent) schauen sich deinen Code an, kommentieren, und geben frei. Dann wird der Branch in main gemergt. CI-Tests laufen automatisch.
  7. 7
    Aufräumen
    Der gemergde Branch wird gelöscht – er wird nicht mehr gebraucht. Die Commits bleiben in main für immer.

Was passiert bei Merge-Konflikten?

Wenn zwei Branches dieselbe Zeile einer Datei unterschiedlich geändert haben, entsteht ein Merge-Konflikt. Git kann nicht automatisch entscheiden, welche Version korrekt ist. Dann muss eine Person (oder ein Agent mit menschlicher Führung) das klären:

Der Ablauf:
  1. Merge wird versucht → Konflikt erkannt
  2. Git markiert die konfliktierenden Stellen in der Datei (ggf. mehrere)
  3. Eine Person schaut sich die Unterschiede an und entscheidet: welche Version ist richtig?
  4. Die Markierungen werden entfernt, die richtige Version bleibt
  5. Neuer Commit mit der gelösten Version (ein "Merge-Commit")
Konflikte vermeiden – die beste Strategie: Häufig synchronisieren (regelmäßig pull) und in kleinen, überschaubaren Einheiten arbeiten. Je länger ein Branch lebt, ohne mit main synchronisiert zu werden, desto größer und komplizierter werden Konflikte.
🔧 Im IT-Systemhaus: geteiltes Inventarisierungs-Tool Du und Kollege Max entwickelt gemeinsam ein Tool zur Inventarisierung von Kundenhardware. Max schreibt die PowerShell-Logik (feature/ps-abfrage), du baust das HTML-Dashboard (feature/dashboard). Beide arbeiten parallel ohne Konflikte. Wenn Max fertig ist, macht er einen PR – du reviewst, ob seine Datenstruktur zu deinem Dashboard passt. Erst dann wird gemergt.

5

Pull Requests & Merge-Strategien

Wenn ein Feature fertig ist und ein Pull Request genehmigt wurde, kommt der Merge. Git bietet drei verschiedene Strategien, um den Feature-Branch in main zu integrieren. Jede hat ihre Stärken.

1️⃣ Merge Commit

Was passiert: Ein spezieller "Merge-Commit" wird erstellt, der zeigt, dass ein Branch zusammengeführt wurde.

* Merge Commit |\ | * Feature Commit 2 | * Feature Commit 1 |/ * main Commit
✅ Vorteile:
  • Beide Feature-Commits bleiben sichtbar
  • Zeigt klar, dass ein Branch existierte
  • Nachvollziehbar für Code-Archäologie
  • Ideal für große Features
❌ Nachteile:
  • main wird länger
  • Tree-Struktur komplexer

2️⃣ Squash & Merge

Was passiert: Alle Feature-Commits werden zu 1 Commit kombiniert und in main eingefügt.

* Squash Commit (enthält Commit 1+2) * main Commit
✅ Vorteile:
  • main bleibt kurz und linear
  • Jedes Feature = 1 übersichtlicher Commit
  • Perfekt für kleine Fixes/Features
  • Saubere, lesbare History
❌ Nachteile:
  • Zwischenschritte nicht mehr sichtbar
  • Einzelne Entwicklungsschritte verloren
  • Aber: ~30 Tage im reflog erreichbar

3️⃣ Rebase & Merge

Was passiert: Feature-Commits werden auf den aktuellen main "umgesetzt", als wäre alles direkt dort entwickelt worden.

* Feature Commit 2 * Feature Commit 1 * main Commit
✅ Vorteile:
  • Commits bleiben sichtbar
  • Lineare, saubere History
  • Keine Merge-Commits
❌ Nachteile:
  • Verbirgt, dass ein Branch existierte
  • Täuscht linearen Entwicklungsprozess vor
  • Kann bei Teams zu Verwirrung führen

Welche Strategie wann?

Im IT-Systemhaus – praktisch:
  • Squash & Merge: Kleine Kundenprojekte, Bug-Fixes, einfache Features (z.B. "Fix typo", "Add footer link")
  • Merge Commit: Große Features, komplexe Entwicklung mit mehreren Steps (z.B. "Implement complete inventory system")
  • Rebase & Merge: Eher selten – nur wenn das Team eine super-lineare History möchte (Enterprise-Standard)

Praktisches Beispiel

🔧 Im IT-Systemhaus

Kunde Müller GmbH – Monitoring-Skript:

Du arbeitest 3 Tage an der CPU-Temperatur-Funktion (3 Commits: Basis, Parsing, Fehlerbehandlung). Beim Merge wählst du Merge Commit – so kann man später nachvollziehen, dass die 3 Schritte zusammengehören.

Dein Kollege verbessert schnell die Button-Farbe (1 Commit). Beim Merge wählt er Squash & Merge – main bleibt übersichtlich.

Die Merge-Strategie wählen – auf GitHub.com

Wenn du einen PR auf GitHub.com mergst, klickst du auf den Merge-Button und siehst ein Dropdown-Menü mit drei Optionen:

  • Create a merge commit – Die Commits bleiben erhalten, ein spezieller "Merge-Commit" zeigt, dass ein Branch zusammengefügt wurde (Default, gut für große Features)
  • Squash and merge – Alle Commits werden zu einem kombiniert (gut für kleine Fixes)
  • Rebase and merge – Commits werden auf den aktuellen main "umgesetzt" (eher selten, sehr linear)
💡 Tipp: Der PR-Author (z.B. ein KI-Agent) kann in der PR-Beschreibung eine Empfehlung geben: „Please use Squash & Merge for this PR". Dann weißt du, was der Agent sich wünscht – und du kannst es bestätigen oder eine andere Strategie wählen. Die finale Entscheidung liegt immer bei dir.

Best Practices – sauber arbeiten mit Git

Aussagekräftige Commit-Nachrichten

Eine Commit-Message beschreibt das Warum und Was, nicht nur die Liste der Dateien. Schreib im Imperativ: "Fix", "Add", "Update" statt "Fixed", "Added".

Gut: „Fix crash when profile photo is missing" oder „Add dark-mode support"

Schlecht: „fix stuff" oder „changes"

Eine zukünftige Person (oder du selbst in 3 Monaten) wird dir danken, wenn die History verständlich ist.

Die richtige Commit-Granularität

Ein Commit = eine logische, abgeschlossene Einheit. Nicht zu groß (ganze Features), nicht zu klein (einzelne Variablennamen).

  • ✅ Gut: Funktion fertig und getestet, Bugfix abgeschlossen, Refactoring-Phase fertig
  • ✅ Ideal: Lieber 5 kleine Commits als 1 Mega-Commit
  • ❌ Zu groß: Mehrere unabhängige Features in einem Commit
  • ❌ Zu klein: Eine Variable umbenennen, dann committen – das kann zusammengehören

Konsistente Branch-Namen

Ein standardisiertes Benennungsschema macht die Übersicht einfacher – für Menschen und Tools.

Standard-Pattern:
feature/funktionalität, fix/bug-beschreibung, release/version

Beispiele: feature/user-auth, fix/login-error, hotfix/security-patch

main ist heilig

main sollte immer in einem stabilen, produktiven Zustand sein. Niemals direkt auf main entwickeln (außer extrem seltene Notfälle).

  • ✅ Regel: Jede Änderung → Feature-Branch → PR → Review → Merge
  • ✅ Sicherheit: Wenn main kaputt ist, ist die Seite kaputt
  • ❌ Nie: Direkte Commits auf main
  • 💡 Tipp: GitHub kann das erzwingen (Branch-Protection-Rules)

.gitignore – was nicht rein darf

Certain files should never go into version control: Passwörter (.env), Build-Artefakte, Logs, lokale Einstellungen, große Dependencies.

Typische Kandidaten:
.env (Secrets), Logs/, node_modules/, *.log, .DS_Store

Die .gitignore-Datei ist selbst teil des Repos – alle Entwickler nutzen die gleichen Regeln.

Synchronisiere regelmäßig

Der längere dein Branch existiert, ohne mit main synchronisiert zu werden, desto größer und komplizierter werden potenzielle Merge-Konflikte.

  • ✅ Ideal: Jeden Tag oder mindestens jeden zweiten Tag pullen
  • ✅ Präventiv: Bevor du anfängst zu arbeiten, kurz pullen
  • ❌ Problematisch: Branch nach 2 Wochen ohne Sync mergen – große Konflikte garantiert
Das wichtigste Prinzip: Git ist dein Sicherheitsnetz – nicht dein Feind. Je öfter du commitest und pushst, desto weniger kann schiefgehen. Kein Commit ist in Stein gemeißelt: mit git revert oder git reset lässt sich fast jede Aktion rückgängig machen.

Windows-Werkzeuge – womit arbeitest du?

Empfohlen

VS Code + GitLens Extension

Visual Studio Code hat eingebaute Git-Unterstützung (Source Control Tab). Mit der Extension GitLens siehst du direkt im Code, wer, wann und warum etwas geändert hat (Git Blame). Das integrierte Terminal erlaubt git-Befehle direkt zu nutzen.

Installation: VS Code Extension-Markt → „GitLens"

Für Einsteiger

GitHub Desktop

Eine grafische Oberfläche speziell für Git – kein Terminal nötig. Zeigt Änderungen visuell (grün = hinzugefügt, rot = entfernt), macht Commits, Pushs und PRs per Klick. Ideal zum Einstieg in Git ohne Kommandozeilen-Angst.

Download: desktop.github.com

Für Profis

Terminal / PowerShell

Git unter Windows installieren (git-scm.com), dann sind alle Git-Befehle in PowerShell und CMD verfügbar. Am flexibelsten und portabel. VS Code hat ein eingebautes Terminal – damit kannst du direkt darin arbeiten.

Vorteil: Volle Kontrolle, kein UI-Umweg, perfekt für Automation und KI-Agenten

Nützlich

Windows Terminal

Ein moderner Ersatz für die alte Windows-Konsole. Unterstützt mehrere Tabs, PowerShell, CMD und WSL in einem Fenster. Besseres Aussehen, bessere Performance, Standard unter Windows 11.

Download: Microsoft Store → „Windows Terminal"

Empfohlenes Setup für Windows-Entwickler: Git für Windows installieren → VS Code installieren → GitLens-Extension aktivieren → fertig. Das Terminal in VS Code ist direkt nutzbar.

Was kannst du mit welcher Oberfläche machen?

GitHub.com (Web)

Perfekt für: Pull Requests erstellen, reviewen, mergen; kleine Dateien editieren; Branches verwalten

  • ✅ PR eröffnen, reviewen, kommentieren
  • ✅ Dateien direkt im Browser editieren
  • ✅ Commits direkt auf GitHub erstellen
  • ✅ Branches erstellen, löschen, zusammenführen
  • ✅ Merge-Strategie wählen (Merge Commit, Squash, Rebase)
  • ❌ Komplexe Merges mit Konflikten lösen

GitHub Desktop

Perfekt für: Visuelle Arbeit mit Branches, Diffs anschauen, einfache Commits

  • ✅ Branches wechseln und erstellen
  • ✅ Änderungen visuell sehen (grün/rot)
  • ✅ Commits lokal erstellen
  • ✅ Pushen und Pullen
  • ✅ Einfache Merges
  • ⚠️ Merge-Konflikte sind möglich, aber kann kompliziert werden

GitHub Mobile App

Perfekt für: Code-Reviews, PRs kommentieren, unterwegs arbeiten

  • ✅ PRs und Issues anschauen
  • ✅ Reviews und Kommentare hinzufügen
  • ✅ Branches mergen (im Notfall)
  • ✅ Notifications sehen
  • ❌ Komplexe Entwicklung
  • ❌ Datei-Bearbeitung

Terminal / PowerShell

Perfekt für: Alles, Automation, KI-Agenten, volle Kontrolle

  • ✅ 100% der Git-Funktionalität
  • ✅ Merge-Konflikte effizient lösen
  • ✅ Complex workflows
  • ✅ Automation und Scripts
  • ✅ Perfekt für Agenten
  • ⚠️ Steilere Lernkurve
Praktisch im Alltag:
  • Kleine Änderungen: GitHub.com Web (schnell editieren, committen, fertig)
  • Visuelle Arbeit: GitHub Desktop (Branches sehen, Diffs vergleichen)
  • Review unterwegs: Mobile App (PR kommentieren, im Zug freigeben)
  • Komplexe Workflows: Terminal (oder Agent mit voller Kontrolle)
  • Automatisierung: Immer Agenten (sie können via Terminal alles machen)

Mit KI-Agenten arbeiten – wie Claude Code Git nutzt

Wenn du mit einem KI-Agenten wie Claude Code arbeitest, interagiert dieser direkt mit Git in deinem Repository – ähnlich wie ein Kollege, der per Fernzugriff am selben Projekt arbeitet.

Was passiert technisch im Hintergrund?

GitHub-Repository (origin) | | git clone (beim Session-Start) ↓ Remote Sandbox (Cloud-Container) ├── Working Directory (geklontes Repo) ├── .git/ (vollständige History) └── Claude Code (hat Zugriff auf alle git-Befehle) | | git add, git commit, git push ↓ GitHub-Repository (origin) ← Änderungen landen hier

Was der Agent tun kann

  • Branches erstellen und wechseln
  • Dateien lesen, bearbeiten, erstellen
  • Commits mit sinnvollen Nachrichten erstellen
  • Branches pushen
  • Mergen und reverten
  • Git-Historie analysieren
  • Pull Requests öffnen mit Beschreibung

Was der Agent nicht tut (ohne Erlaubnis)

  • Force-Push auf main
  • Destructive operations ohne Nachfragen
  • Direkt auf main committen (normalerweise)
  • Branch-Protection-Regeln umgehen
  • Automatisch mergen (das machst du)
Wichtig zu wissen: Git-Hooks im Agenten Lokale Git-Hooks (z.B. ein Pre-Commit-Hook, der Timestamps automatisch aktualisiert) laufen möglicherweise nicht in der Remote-Sandbox – der Hook liegt zwar in .git/hooks/, aber Hooks werden bei einem frischen Clone oft nicht automatisch ausgeführt. Das erklärt, warum manche automatischen Aktionen (wie Timestamp-Updates) beim Agenten-Commit ausbleiben können.

Wie du sinnvoll mit einem Agenten zusammenarbeitest

  1. ✓
    Klare, konkrete Aufgaben – Der Agent kann mit vagen Anforderungen nichts anfangen. „Ändere die Farbe der h2-Headlines auf Dunkelrot und füge einen Stern hinzu" ist 100x besser als „mach die Seite schöner".
  2. ✓
    Branch-Strategie festlegen – Sag dem Agenten, ob er auf einen Feature-Branch arbeiten soll oder direkt auf main. Was sind deine Policies?
  3. ✓
    Erwarte Nachfragen bei risikanten Aktionen – Ein guter Agent sollte nicht einfach alle Befehle ausführen. Er sollte nachfragen, bevor er etwas Destruktives tut (Force-Push, Datenlöschung).
  4. ✓
    Überprüfe, was gemacht wurde – Schau dir die Commits an. Sind die Messages sinnvoll? Ist die Granularität richtig? Die Git-Historie ist dein Audit-Trail.
  5. ✓
    PR-Review ist immer noch Menschenwerk – Der Agent kann einen PR öffnen, aber die finale Freigabe sollte ein Mensch machen – besonders bei kritischen Änderungen.

Mehrere Projekte – mehrere Repositories

In einem Team arbeiten oft mehrere Personen an verschiedenen Projekten gleichzeitig. Jedes Projekt hat sein eigenes Repository.

Typische Struktur lokal

Wie Git wirklich funktioniert – die technische Basis

Viele denken, Git speichere einfach Dateiversionen wie ein Papierkorb. In Wirklichkeit ist Git ein inhaltsadressierbares Dateisystem – eine elegante Objektdatenbank. Das Verständnis davon erklärt, warum Git so mächtig ist und warum man immer zu einem früheren Stand zurück kann.

Der .git-Ordner – das Herz des Repositories

In jedem Git-Projekt gibt es einen versteckten .git-Ordner. Hier speichert Git alles, was es braucht.

Struktur (vereinfacht):

  • objects/ – Die echte Datenbank. Hier liegen alle Commits, Dateien und Verzeichnisse komprimiert und unveränderbar. Jedes Objekt hat einen SHA-1-Hash als Name.
  • refs/heads/ – Die Branches. Jede Datei hier enthält einfach den SHA eines Commits (z.B. main zeigt auf Commit abc123...).
  • HEAD – Ein Zeiger, der sagt, welcher Branch aktiv ist (z.B. „ref: refs/heads/main").
  • config – Deine lokalen Git-Einstellungen (z.B. welcher Remote als origin gilt).
  • hooks/ – Scripts, die Git automatisch bei bestimmten Ereignissen ausführt (z.B. vor jedem Commit).
Kernidee:

Die Dateien in deinem Projektordner (z.B. docs/index.html) sind nur eine Anzeige des aktuellen Commits – ein „Checkout" aus der Datenbank. Die echten Daten, alle älteren Versionen, alle Branches, die ganze Geschichte: alles liegt komprimiert und gesichert in .git/objects/.

Die 4 Git-Objekttypen

Blob

Der rohe Dateiinhalt – z.B. der gesamte Text einer JavaScript-Datei. Kein Dateiname, kein Pfad – nur reiner Inhalt.

Tree

Eine Verzeichnisliste: zeigt auf Blobs (Dateien) und andere Trees (Unterordner). Entspricht einem Ordner in deinem Projekt.

Commit

Ein Snapshot: zeigt auf einen Tree (Zustand des Projekts), den Parent-Commit, Autor, Zeitstempel und Beschreibung.

Tag

Ein benannter Zeiger auf einen bestimmten Commit – z.B. v1.0.0 für eine Release-Version.

SHA-1-Hashing – der Fingerabdruck

Jedes Objekt wird durch den SHA-1-Hash seines Inhalts adressiert (z.B. c82eab66b20d06953e...). Ändert sich auch nur ein einziges Zeichen, entsteht ein komplett neuer Hash. Das garantiert:

  • 🔒 Integrität – Du kannst sofort erkennen, ob ein Objekt verändert wurde
  • ⚡ Effizienz – Identische Dateien werden nur einmal gespeichert
  • ⏮️ Unveränderlichkeit – Alles bleibt erhalten, nichts wird überschrieben

Der Commit-Graph (DAG)

Commits bilden einen gerichteten azyklischen Graphen (Directed Acyclic Graph). Jeder Commit zeigt auf seinen Vorgänger. Branches sind dabei nur leichtgewichtige Zeiger – eine simple Textdatei mit einem SHA-Hash:

A ← B ← C ← F (main) \ / D ← E (feature/login) A, B, C... = einzelne Commits (unveränderliche Snapshots) F = Merge-Commit (hat zwei Parents: C und E) main = Zeiger auf F (nur ein SHA-Hash, kein echter "Zweig") feature = Zeiger auf E
Das erklärt, warum man immer zurück kann: Da Commits nie gelöscht werden (nur Zeiger bewegen sich), ist jeder frühere Zustand des Projekts dauerhaft im .git/objects/-Ordner gespeichert und erreichbar.

Staging Area

Eine Zwischenebene zwischen deinen Dateien (Working Directory) und dem nächsten Commit. Du entscheidest bewusst, welche Änderungen reingehören – nicht alle.

Du legst Sachen in einen Karton, bevor du ihn abschickst – du kannst noch wählen, was reinkommt und was liegen bleibt.

Praktisch: Du kannst 5 Dateien geändert haben, aber nur 3 davon in den nächsten Commit nehmen. Die anderen Änderungen bleiben lokal.

HEAD

Ein spezieller Zeiger, der angibt, wo du gerade „bist" – welcher Branch oder Commit aktiv ist. Technisch ist HEAD nur eine Textdatei in .git/.

Die „Sie sind hier"-Markierung auf einer Karte. Sie sagt: von hier aus werden neue Commits gemacht.

Wichtig: HEAD zeigt auf einen Branch oder einen Commit. Wenn du einen Branch wechselst, wechselt auch HEAD automatisch.

Fork

Eine eigene, komplette Kopie eines fremden Repositories auf GitHub-Ebene – nicht nur lokal, sondern auf deinem Account. Typisch bei Open-Source-Beiträgen oder wenn du ein Projekt unabhängig weiterentwickeln willst.

Du kopierst das ganze Projekt in deinen GitHub-Account, damit du frei daran arbeiten kannst – ohne das Original zu beeinflussen.

Workflow: Fork → Clone lokal → Änderungen → PR zum Original-Repo (um deine Verbesserungen zu teilen)

Revert

Erzeugt einen neuen Commit, der die Änderungen eines früheren Commits exakt rückgängig macht. Das ist die sichere „Undo"-Methode für bereits gepushte Commits.

Du machst eine Änderung rückgängig, aber Git dokumentiert trotzdem, dass es sie gab und dass du sie rückgängig gemacht hast. Volle Transparenz.

Unterschied zu Reset: Revert ist sicher (dokumentiert alles), Reset ist destruktiv (löscht Commits aus der History).

Lokal vs. Remote – die zwei Welten

Git ist ein verteiltes Versionskontrollsystem: Jeder Entwickler hat eine vollständige Kopie des Repositories auf seinem PC. Es gibt keinen zentralen Server, der unbedingt gebraucht wird – GitHub ist bequem, aber kein Muss.

Dein PC (lokal) GitHub (remote) +-------------------------------+ +---------------------------+ | | | | | Working Directory | | origin/main | | (deine sichtbaren Dateien) | | origin/feature/... | | | | | | | git add | push | (alle gepushten Commits | | | | ====> | und Branches) | | Staging Area | | | | | | <==== | | | git commit | pull | | | | | fetch | | | Local Repository (.git/) | | | | (vollständige History!) | | | +-------------------------------+ +---------------------------+

Lokal – was du ohne Internet tun kannst

  • Commits erstellen (git commit)
  • Branches anlegen und wechseln
  • Die gesamte Historie einsehen
  • Dateien vergleichen (git diff)
  • Commits rückgängig machen (git revert)

Remote – was eine Verbindung braucht

  • Commits hochladen (git push)
  • Commits herunterladen (git pull / fetch)
  • Pull Requests erstellen
  • Branches mit Teamkollegen teilen
  • CI/CD-Pipelines auslösen
Praxistipp: Committe regelmäßig lokal, auch wenn du noch nicht pushen willst. Lokale Commits kosten nichts und sichern deinen Fortschritt – gepusht wird, wenn du bereit bist.

Der tägliche Workflow – allein und im Team

Der bekannteste Workflow ist der Feature-Branch-Workflow: Jedes neue Feature oder jeder Bugfix bekommt einen eigenen Branch. Nur fertige, geprüfte Änderungen kommen in main. Das funktioniert allein genauso wie im Team.

Der Standard-Ablauf

  1. 1
    Neuesten Stand holen
    git pull origin main – bevor du anfängst, immer synchronisieren.
  2. 2
    Eigenen Branch erstellen
    git checkout -b feature/login-button – du arbeitest isoliert, ohne main zu berühren.
  3. 3
    Coden und committen
    Regelmäßig kleine, sinnvolle Commits machen. Nicht alles auf einmal.
  4. 4
    Branch pushen
    git push -u origin feature/login-button – der Branch ist jetzt auf GitHub sichtbar.
  5. 5
    Pull Request erstellen
    Auf GitHub: PR von deinem Branch in main. Kollegen können reviewen und kommentieren.
  6. 6
    Review & Merge
    Nach Freigabe wird der Branch in main gemergt. CI-Tests laufen automatisch.
  7. 7
    Branch löschen
    Gemergter Branch kann gelöscht werden – er wird nicht mehr gebraucht.

Was tun bei Merge-Konflikten?

Wenn zwei Personen dieselbe Zeile einer Datei geändert haben, entsteht ein Merge-Konflikt. Git markiert die konfliktierenden Stellen:

<<<<<<< HEAD (deine Version) color: #8B0000; ======= color: #0078D4; >>>>>>> feature/neue-farbe (andere Version)

Du entscheidest manuell, welche Version (oder eine Kombination) korrekt ist, entfernst die Markierungen, und erstellst dann einen neuen Commit.

Konflikte vermeiden: Häufig pullen (git pull) und in kleinen, überschaubaren Einheiten arbeiten. Je länger ein Branch lebt, ohne mit main synchronisiert zu werden, desto größer werden Konflikte.

Max schreibt die PowerShell-Logik (feature/ps-abfrage), du baust das HTML-Dashboard (feature/dashboard). Beide arbeiten parallel ohne Konflikte. Wenn Max fertig ist, macht er einen PR – du reviewst, ob seine Datenstruktur zu deinem Dashboard passt. Erst dann wird gemergt.

Pull Requests & Merge-Strategien

Wenn ein Feature fertig ist und ein Pull Request genehmigt wurde, kommt der Merge. Git bietet drei verschiedene Strategien, um den Feature-Branch in main zu integrieren. Jede hat ihre Stärken.

1️⃣ Merge Commit

Was passiert: Ein spezieller "Merge-Commit" wird erstellt, der zeigt, dass ein Branch zusammengeführt wurde.

* Merge Commit |\ | * Feature Commit 2 | * Feature Commit 1 |/ * main Commit
✅ Vorteile:
  • Beide Feature-Commits bleiben sichtbar
  • Zeigt klar, dass ein Branch existierte
  • Nachvollziehbar für Code-Archäologie
  • Ideal für große Features
❌ Nachteile:
  • main wird länger
  • Tree-Struktur komplexer

2️⃣ Squash & Merge

Was passiert: Alle Feature-Commits werden zu 1 Commit kombiniert und in main eingefügt.

* Squash Commit (enthält Commit 1+2) * main Commit
✅ Vorteile:
  • main bleibt kurz und linear
  • Jedes Feature = 1 übersichtlicher Commit
  • Perfekt für kleine Fixes/Features
  • Saubere, lesbare History
❌ Nachteile:
  • Zwischenschritte nicht mehr sichtbar
  • Einzelne Entwicklungsschritte verloren
  • Aber: ~30 Tage im reflog erreichbar

3️⃣ Rebase & Merge

Was passiert: Feature-Commits werden auf den aktuellen main "umgesetzt", als wäre alles direkt dort entwickelt worden.

* Feature Commit 2 * Feature Commit 1 * main Commit
✅ Vorteile:
  • Commits bleiben sichtbar
  • Lineare, saubere History
  • Keine Merge-Commits
❌ Nachteile:
  • Verbirgt, dass ein Branch existierte
  • Täuscht linearen Entwicklungsprozess vor
  • Kann bei Teams zu Verwirrung führen

Welche Strategie wann?

Im IT-Systemhaus – praktisch:
  • Squash & Merge: Kleine Kundenprojekte, Bug-Fixes, einfache Features (z.B. "Fix typo", "Add footer link")
  • Merge Commit: Große Features, komplexe Entwicklung mit mehreren Steps (z.B. "Implement complete inventory system")
  • Rebase & Merge: Eher selten – nur wenn das Team eine super-lineare History möchte (Enterprise-Standard)

Praktisches Beispiel

🔧 Im IT-Systemhaus

Kunde Müller GmbH – Monitoring-Skript:

Du arbeitest 3 Tage an der CPU-Temperatur-Funktion (3 Commits: Basis, Parsing, Fehlerbehandlung). Beim Merge wählst du Merge Commit – so kann man später nachvollziehen, dass die 3 Schritte zusammengehören.

Dein Kollege verbessert schnell die Button-Farbe (1 Commit). Beim Merge wählt er Squash & Merge – main bleibt übersichtlich.

In der Git App / GitHub

Wenn du einen PR mergst, siehst du einen Dropdown-Button:

▼ Merge pull request ☐ Create a merge commit ← Default ☐ Squash and merge ← Wähle diese für kleine Änderungen ☐ Rebase and merge ← Selten nötig

Tipp: Der PR-Author (z.B. ein KI-Agent) kann in der PR-Beschreibung explizit empfehlen, welche Strategie zu wählen ist. Dann weißt du, was gewünscht ist.

Best Practices – sauber arbeiten mit Git

Gute Commit-Nachrichten

Eine gute Nachricht beschreibt das Warum, nicht nur das Was. Im Imperativ schreiben.

# ❌ Schlecht git commit -m "fix stuff" git commit -m "changes" # ✅ Gut git commit -m "Fix crash when profile photo is missing" git commit -m "Add dark-mode support to dashboard"

Wann committen?

Ein Commit = eine logische Einheit. Nicht zu groß, nicht zu klein. Lieber öfter als zu selten.

  • ✅ Funktion fertig und getestet
  • ✅ Bugfix abgeschlossen
  • ✅ Bevor du ein Experiment startest
  • ❌ Mitten in einer halbfertigen Funktion
  • ❌ Dateien, die nicht dazugehören

Branch-Benennung

Konsistente Namen machen die Übersicht einfacher – besonders im Team.

feature/user-authentication feature/dark-mode fix/login-crash fix/typo-in-header release/v2.0.0 hotfix/critical-security-patch

main immer stabil halten

main sollte immer deploybar sein. Nie direkt auf main entwickeln.

  • ✅ Feature-Branches für jede Änderung
  • ✅ PR-Review vor dem Merge
  • ✅ Automatische Tests (CI) bei jedem PR
  • ❌ Direkte Commits auf main (außer Notfälle)

.gitignore richtig einsetzen

Bestimmte Dateien sollen nie in Git landen: Passwörter, Build-Artefakte, lokale Einstellungen.

# Typische .gitignore-Einträge: node_modules/ .env *.log dist/ .DS_Store Logs/

Regelmäßig pullen

Je länger dein Branch nicht mit main synchronisiert wurde, desto größer werden Merge-Konflikte.

# Am Anfang jedes Arbeitstages: git pull origin main # Oder den eigenen Branch aktualisieren: git fetch origin git rebase origin/main
Das wichtigste Prinzip: Git ist dein Sicherheitsnetz – nicht dein Feind. Je öfter du commitest und pushst, desto weniger kann schiefgehen. Kein Commit ist in Stein gemeißelt: mit git revert oder git reset lässt sich fast jede Aktion rückgängig machen.

Viele denken, Git speichere einfach Dateiversionen wie ein Papierkorb. In Wirklichkeit ist Git ein inhaltsadressierbares Dateisystem – eine elegante Objektdatenbank. Das Verständnis davon erklärt, warum Git so mächtig ist und warum man immer zu einem früheren Stand zurück kann.

Der .git-Ordner – das Herz des Repositories

In jedem Git-Projekt gibt es einen versteckten .git-Ordner. Darin liegt alles:

.git/ ├── objects/ # Die eigentliche Datenbank – alle Objekte │ ├── 1a/ │ │ └── b4c5d9... # Ein Objekt (Blob: Inhalt von index.html) │ ├── 3f/ │ │ └── e7a2b1... # Ein Objekt (Tree: Verzeichnis /docs) │ └── c8/ │ └── 2eab66... # Ein Objekt (Commit: "Add stars") ├── refs/ │ └── heads/ │ ├── main # Enthält: c82eab6... (nur ein SHA-Zeiger!) │ └── feature # Enthält: 5b96f9f... └── HEAD # Enthält: ref: refs/heads/main
Wichtige Erkenntnis: Die Dateien in deinem Projektordner (z.B. docs/index.html) sind nur eine Ansicht des aktuellen Commits. Die echten Daten liegen komprimiert im .git/objects/-Ordner.

Die 4 Git-Objekttypen

Blob

Der rohe Dateiinhalt – z.B. der gesamte Text einer JavaScript-Datei. Kein Dateiname, kein Pfad – nur reiner Inhalt.

Tree

Eine Verzeichnisliste: zeigt auf Blobs (Dateien) und andere Trees (Unterordner). Entspricht einem Ordner in deinem Projekt.

Commit

Ein Snapshot: zeigt auf einen Tree (Zustand des Projekts), den Parent-Commit, Autor, Zeitstempel und Beschreibung.

Tag

Ein benannter Zeiger auf einen bestimmten Commit – z.B. v1.0.0 für eine Release-Version.

SHA-1-Hashing – der Fingerabdruck

Jedes Objekt wird durch den SHA-1-Hash seines Inhalts adressiert (z.B. c82eab66b20d06953e...). Ändert sich auch nur ein einziges Zeichen, entsteht ein komplett neuer Hash. Das garantiert:

  • 🔒 Integrität – Du kannst sofort erkennen, ob ein Objekt verändert wurde
  • ⚡ Effizienz – Identische Dateien werden nur einmal gespeichert
  • ⏮️ Unveränderlichkeit – Alles bleibt erhalten, nichts wird überschrieben

Der Commit-Graph (DAG)

Commits bilden einen gerichteten azyklischen Graphen (Directed Acyclic Graph). Jeder Commit zeigt auf seinen Vorgänger. Branches sind dabei nur leichtgewichtige Zeiger – eine simple Textdatei mit einem SHA-Hash:

A ← B ← C ← F (main) \ / D ← E (feature/login) A, B, C... = einzelne Commits (unveränderliche Snapshots) F = Merge-Commit (hat zwei Parents: C und E) main = Zeiger auf F (nur ein SHA-Hash, kein echter "Zweig") feature = Zeiger auf E
Das erklärt, warum man immer zurück kann: Da Commits nie gelöscht werden (nur Zeiger bewegen sich), ist jeder frühere Zustand des Projekts dauerhaft im .git/objects/-Ordner gespeichert und erreichbar.

Weitere Begriffe – für Fortgeschrittene

Diese Begriffe begegnen dir, sobald du tiefer einsteigst. Für den täglichen Workflow reichen die 8 Kernbegriffe aus Teil 1.

Staging Area

Eine Zwischenebene zwischen Working Directory und Commit. Du entscheidest bewusst, welche Änderungen in den nächsten Commit kommen.

Du legst Sachen in einen Karton, bevor du ihn abschickst – du kannst noch wählen, was reinkommt.

# Nur diese Datei für den Commit vorbereiten git add docs/styles.css # Alle geänderten Dateien vorbereiten git add . # Zeigen, was in der Staging Area liegt git status

HEAD

Ein spezieller Zeiger, der angibt, wo du gerade „bist" – welcher Branch oder Commit aktiv ist. .git/HEAD ist schlicht eine Textdatei.

Die „Sie sind hier"-Markierung auf einer Karte.

# HEAD zeigt auf den aktuellen Branch cat .git/HEAD # Ausgabe: ref: refs/heads/main # Commits relativ zu HEAD ansehen git log HEAD~3..HEAD

Fork

Eine eigene Kopie eines fremden Repositories auf GitHub – auf Server-Ebene, nicht nur lokal. Typisch bei Open-Source-Beiträgen.

Du legst eine Kopie des ganzen Projekts in deinem eigenen GitHub-Account an und arbeitest dort.

Ablauf: Fork auf GitHub → lokaler Clone → Änderungen → PR an das Original-Repo

Revert

Erzeugt einen neuen Commit, der einen bestimmten früheren Commit exakt rückgängig macht. Die sicherste „Undo"-Methode für bereits gepushte Commits.

Du machst eine Änderung rückgängig, aber das Protokoll zeigt trotzdem, dass es sie gab.

# Einen bestimmten Commit rückgängig machen git revert c82eab6 # Ohne automatischen Commit (manuell bestätigen) git revert --no-edit c82eab6

Praxis-Durchlauf: Ein Kundenprojekt von A bis Z

Szenario: Kunde Bauer GmbH braucht ein PowerShell-Skript, das täglich Festplattenplatz prüft und eine Log-Datei schreibt. Das Projekt soll sauber mit Git verwaltet werden – auch wenn später Änderungen kommen.

📁

Schritt 1: Projekt starten, Git initialisieren

Neues Projekt braucht ein Git-Repository als Fundament. Das ist einmalig am Anfang:

  • Neuer Ordner für das Projekt anlegen (z.B. kunde-bauer-diskcheck)
  • Git initialisieren – ein versteckter .git-Ordner wird angelegt (die Archivdatenbank)
  • Ein neues, leeres Repository auf GitHub (oder GitLab, etc.) erstellen
  • Lokales Repo mit dem Remote verknüpfen, damit du pushen kannst

Was der Agent hier tut: Er erstellt die Ordner-Struktur, initialisiert Git lokal und verbindet alles mit GitHub.

✏️

Schritt 2: Erste Implementierung, erster Commit

Die Grundstruktur des Skripts ist da. Es läuft, aber nicht alles ist perfekt. Das ist ein sinnvoller Zwischenstand:

  • Das PowerShell-Skript wird geschrieben und getestet
  • Eine .gitignore-Datei wird angelegt – sie sagt Git, dass Logs nicht ins Repo gehören
  • Beide Dateien werden vorbereitet (Staging Area)
  • Ein sauberer Commit wird gemacht: „Add initial disk check script with basic log output"
  • Der Branch wird auf GitHub gepusht – der Kunde und das Team sehen die Arbeit

Was du lernst: Commits sind nicht „final" – das erste Commit ist erst der Anfang. Es dokumentiert den Startpunkt für alle zukünftigen Änderungen.

🔧

Schritt 3: Neues Feature – E-Mail-Benachrichtigung

Kunde möchte eine neue Funktion: E-Mail-Benachrichtigung, wenn der freie Speicher unter 10 % fällt. Das ist ein neues Feature:

  • Ein neuer Branch wird erstellt (z.B. feature/email-benachrichtigung) – damit bleibt main stabil
  • Die neue Funktion wird auf diesem Branch entwickelt, getestet und mehrmals committed
  • Der Branch wird auf GitHub gepusht
  • Ein Pull Request wird eröffnet – „Schau dir meine E-Mail-Funktion an und gib mir Feedback"
  • Kollege Max (oder ein Agent mit Manual-Review) schaut sich den Code an, stellt Fragen, gibt Feedback

Was passiert: Das Team sieht genau, was geändert wurde (grün = neu, rot = entfernt). Fehler können noch aufgegriffen werden, bevor das Feature produktiv geht.

🔀

Schritt 4: Review bestätigt, Feature wird zusammengefügt

Max hat die E-Mail-Funktion geprüft und freigegeben. Jetzt wird das Feature in main integriert:

  • Der PR wird als „Approved" markiert
  • Automatische Tests (CI) prüfen, ob alles noch zusammen funktioniert
  • Der PR wird gemergt – die Commits aus dem Feature-Branch werden in main integriert
  • Das System entscheidet die Merge-Strategie: Merge Commit, Squash oder Rebase (abhängig von deiner Konfiguration)
  • Der Feature-Branch wird gelöscht – er wird nicht mehr gebraucht
  • main ist jetzt um ein Feature reicher, dokumentiert, und bereit für den nächsten Deploy

Wichtig: Das Merge ist nicht das Ende – main muss jetzt deployed werden. Bei GitHub Pages ist das automatisch, bei anderen Systemen gibt es einen Deploy-Prozess.

🚨

Schritt 5: Notfall – Skript crasht beim Kunden

Zwei Wochen später: Anruf vom Kunden. Das Skript bricht mit einem Fehler ab, wenn ein Netzwerk-Laufwerk nicht erreichbar ist. Das ist keine neue Funktion, das ist ein Bugfix – und er muss raus:

  • Synchronisiere mit dem neuesten main (pull)
  • Erstelle einen Hotfix-Branch – ähnlich einem Feature-Branch, aber für Bugs (z.B. hotfix/null-check-laufwerk)
  • Finde das Problem und schreibe ein Test, das nachweist, dass der Bug existiert
  • Behebe den Fehler
  • Commit (aussagekräftig: „Fix null reference when network drive is unreachable")
  • Push den Hotfix-Branch und öffne einen PR
  • Review (schnell, weil es ein Notfall ist) und Merge

Das Wichtige: Ein Hotfix ändert nichts an den geplanten Features. Andere Branches (z.B. feature/xyz) werden nicht beeinflusst. Der Hotfix ist eine schnelle Reparatur für main.

🔁

Schritt 6: Neuer Kunde – ähnliches Projekt, andere Anforderungen

Kunde Vogel GmbH möchte das gleiche Skript, aber mit anderen Schwellwerten und einem anderen Log-Pfad. Wie geht man vor?

Es gibt mehrere Strategien – jede hat Pro und Contra:

  • Neues Repository klonen: Der saubere Weg. Du machst ein neues Repo für Vogel, fängst mit dem bestehenden Skript an und entwickelst es weiter. Jeder Kunde hat sein eigenes Repo, seine eigene History, völlig unabhängig.
  • Konfiguration auslagern (empfohlen): Das Skript bleibt im gleichen Repo, aber Schwellwerte und Pfade werden in eine externe Konfigurationsdatei ausgelagert (z.B. config-bauer.json, config-vogel.json). Diese Config-Dateien landen nicht im Repo (.gitignore). Ein Repo, viele Kunden.
  • Branch pro Kunde (nicht empfohlen): Beide Kunden im gleichen Repo auf unterschiedlichen Branches. Das funktioniert nur bei sehr kleinen Unterschieden und wird schnell unübersichtlich, wenn beide Kunden Updates brauchen.

Im IT-Systemhaus: Strategie 1 (neues Repo) ist am saubersten. Strategie 2 (Config auslagern) ist am effizientesten, wenn die Basis-Logik gleich bleibt.

📋

Schritt 7: Das Repository ist die Dokumentation

Nach sechs Monaten tritt ein neuer Kollege ins Team. Er schaut sich das Projekt an – nicht die Emails, nicht die Word-Dokumente, sondern die Git-History:

a3f9c12 Fix null reference when network drive is unreachable 8b2e4a1 Add email alert when disk space below 10% 3c7d9f0 Add initial disk check script with basic log output

Drei Commits. Er kann sofort sehen:

  • Was wurde gemacht: Initiales Skript, dann E-Mail-Funktion, dann ein Bugfix
  • In welcher Reihenfolge: Der 3c7... Commit war zuerst, dann kam 8b2..., dann der Bugfix
  • Wann wurde gemacht: Git speichert Timestamps mit jedem Commit
  • Von wem: Git speichert Autor und Committer
  • Warum wurde gemacht: Die Commit-Message erklärt es (wenn sie gut geschrieben ist)

Das ist die Kraft von Git: Saubere Commits sind deine einzige echte Dokumentation. Kein Word-Dokument, kein „v2_FINAL" im Dateinamen – nur klare, permanente Dokumentation in der History.

Das Muster für jedes Projekt im Systemhaus: Neues Projekt → git init + Remote anlegen → kleine Commits von Anfang an → Features in Branches → Kollege reviewed → Merge → Branch löschen. Egal ob das Projekt 2 Stunden oder 2 Jahre dauert.
