Teil 1 · Die Basis

Git verstehen

Fundamental: Was Git ist, wie es funktioniert, die Kernideen.

1

Was ist Git?

Git ist ein verteiltes Versionskontrollsystem. Das bedeutet:

  • Es speichert nicht nur deine aktuelle Datei, sondern die gesamte Geschichte aller Änderungen
  • Jeder (Mensch oder Agent) hat eine komplette Kopie dieser Geschichte auf seinem PC
  • Es gibt keinen zentralen Server, der alles speichert – alle sind gleichberechtigt
  • GitHub ist einfach ein zentraler Ort, wo alle ihre Arbeit zusammentragen
Warum das wichtig ist: Du kannst jederzeit zu einem früheren Stand zurück. Wenn ein Agent etwas kaputtmacht, ist nichts verloren. Die komplette Geschichte bleibt dokumentiert.
🔧 Im IT-Systemhaus

Du hast für einen Kunden ein Monitoring-Skript geschrieben. Der Agent verbessert es über Monate. Drei Monate später: „Moment, wann wurde Feature X eingefügt?" Mit Git siehst du in einer Sekunde: welcher Commit, von wem, mit welcher Beschreibung. Kein „Dateiname-v3-FINAL-wirklich-final.ps1" nötig.

Die 8 Kernkonzepte

Diese 8 Ideen sind das gesamte Git-Universum. Alles andere baut darauf auf.

Repository

Ein Projektordner mit einem versteckten .git-Verzeichnis – der Archivordner. Hier speichert Git die gesamte Geschichte.

Ein Projekt-Tagebuch. Vorne die aktuelle Seite (deine Dateien), hinten alle bisherigen Einträge (die History).

Clone

Eine komplette Kopie eines Repositories – mit der gesamten Geschichte. Danach hast du alles lokal.

Du bekommst eine Kopie des kompletten Tagebuchs. Du brauchst danach kein Netzwerk mehr, um die History zu sehen.

Commit

Ein unveränderlicher Snapshot deines Projekts – mit Beschreibung, Autor, Zeit. Commits sind die Bausteine deiner Geschichte.

Ein neuer Eintrag im Tagebuch. „Heute habe ich Feature X hinzugefügt." Dieser Eintrag ist permanent und kann nicht verändert werden.

Branch

Ein paralleler Entwicklungsstrang – eine eigene Zeitleiste für ein Feature oder einen Bugfix. Mehrere Branches können gleichzeitig existieren.

Eine Kopie des Tagebuchs, auf der du experimentierst. Das Original bleibt unberührt. Wenn dein Experiment gut geht, fügst du die Änderungen ins Original ein.

Push

Deine lokalen Commits auf GitHub hochladen, damit andere (und Agenten) sie sehen können.

Du schickst dein Tagebuch an die zentrale Stelle, damit alle deine Einträge lesen können.

Pull

Die neuesten Commits von GitHub herunterladen und in deinen lokalen Branch integrieren.

Du holst die neuesten Einträge aus der zentralen Stelle und liest sie in dein Tagebuch ein.

Merge

Zwei Branches zusammenbringen – die Commits aus einem Branch werden in den anderen integriert. Git kombiniert automatisch, bei Konflikten musst du helfen.

Dein Experiment war erfolgreich. Du fügst die Änderungen aus der Kopie ins Original zurück.

Pull Request (PR)

Ein formales „Schau dir meine Arbeit an und gib mir Feedback, bevor ich es in main merge." Ermöglicht Review und Diskussion auf GitHub.

Du reichst dein Experiment zur Qualitätskontrolle ein. Kollegen lesen, kommentieren, und geben freigabe.

Lokal vs. Remote – zwei Welten

Git hat immer zwei Orte:

Lokal (dein PC)

Dein lokales Repository mit der kompletten History. Hier machst du Commits, wechselst Branches, merkst du, ob was kaputt ist. Kein Internet nötig.

Remote (GitHub)

Der zentrale Ort, wo das Team und die Agenten ihre Arbeit teilen. Hier passieren PRs, Code-Reviews, Merges, CI/CD.

Das Wichtigste:

Du brauchst nicht immer Netzwerk. Arbeite lokal, committe lokal, und pushe nur, wenn du deine Arbeit teilen willst. Solange es lokal ist, ist es private Sandbox.

Wie der Agent mit Git arbeitet

Der Agent (z.B. Claude Code) hat Zugriff auf alle Git-Operationen. Er kann:

✅ Der Agent kann:

  • Branches erstellen
  • Dateien ändern, erstellen, löschen
  • Commits mit guten Nachrichten erstellen
  • Branches pushen
  • PRs öffnen mit Beschreibung
  • Mergen (wenn du es freigibst)

❌ Der Agent macht nicht:

  • Force-Push auf main (ohne Genehmigung)
  • Destruktive Dinge ohne zu fragen
  • Direkt auf main committen (normalerweise)
  • Automatisch mergen (das machst du)
Wichtig zu wissen: Der Agent arbeitet in einer Remote-Sandbox (Cloud-Container). Er klont das Repo, macht Änderungen, und pushed zu GitHub. Das ist genau wie wenn ein Kollege remote arbeitet.

Wie du Aufgaben beschreibst

Der Agent kann nur so gut arbeiten, wie du die Aufgabe erklärst. Hier sind die Best Practices:

Sei konkret

Schlecht: „Mach die Seite schöner"

Gut: „Ändere die h2-Headlines auf Dunkelrot (#8B0000) und füge einen Stern-Emoji davor"

Nenne das Ziel

Schlecht: „Update das Skript"

Gut: „Das Monitoring-Skript crasht, wenn ein Netzwerk-Laufwerk nicht erreichbar ist. Implementiere einen Null-Check"

Gib den Kontext

Link zur Datei, Screenshot, Fehler-Output, PR-Nummer – alles, was dem Agenten hilft, schnell zu verstehen, was zu tun ist.

Klär die Branch-Strategie

„Feature-Branch, dann PR nach main" oder „Direkt auf main committen"? Der Agent fragt, wenn unklar, aber sag es lieber vorher.

6

Ergebnisse reviewen – wie du Änderungen prüfst

Der Agent hat die Arbeit gemacht. Jetzt prüfst du sie. Hier sind die Stationen:

  1. 1
    Schau dir den PR auf GitHub an
    Der PR zeigt dir genau, welche Zeilen geändert wurden (rot = weg, grün = neu). Du siehst sofort, ob was unreasonable ist.
  2. 2
    Lies die Commit-Nachrichten
    Sind die aussagekräftig? „Add dark mode" ist gut. „fix" ist schlecht. Das ist wichtig für die Zukunft.
  3. 3
    Teste lokal (optional)
    Wenn es wichtig ist, klone den Branch lokal und teste die Änderungen. Meist ist das GitHub-Preview genug.
  4. 4
    Kommentiere im PR, wenn Feedback nötig ist
    Der Agent liest Kommentare und kann weitere Commits machen. Dann wird der PR automatisch aktualisiert.
  5. 5
    Gib frei (Approve)
    Wenn alles gut aussieht: „Approved" – der Agent oder deine CI/CD kann jetzt mergen.
Trick: Diff-View auf GitHub

Oben rechts im PR kannst du zwischen „Unified" (eine Spalte) und „Split" (zwei Spalten) umschalten. Split ist oft übersichtlicher für große Changes.

Merge-Strategien – wie die Commits zusammengefügt werden

Wenn du einen PR auf GitHub mergst, kannst du zwischen 3 Strategien wählen. Jede hat ein anderes Ergebnis:

1️⃣ Merge Commit

Was passiert: Ein spezieller „Merge-Commit" wird erstellt, der zeigt: hier wurde ein Branch zusammengefügt.

History sieht aus wie: Zwei parallele Zeilen, dann vereinigen sich wieder.

Wann: Große Features mit mehreren Commits, wo man später nachvollziehen will, dass das eine Einheit war.

2️⃣ Squash & Merge

Was passiert: Alle Commits aus dem Feature-Branch werden zu 1 Commit kombiniert.

History sieht aus wie: Eine saubere, lineare Linie. Keine Verzweigungen.

Wann: Kleine Fixes, Single Features, wenn man nicht alle Zwischen-Commits braucht.

3️⃣ Rebase & Merge

Was passiert: Die Commits werden „umgesetzt", als wären sie direkt auf main gemacht worden.

History sieht aus wie: Linear, aber es wird verschleiert, dass ein Branch existierte.

Wann: Eher selten – nur wenn absolute lineare History wichtig ist.

Praktische Empfehlung im IT-Systemhaus:
  • Squash & Merge: Kleine Fixes, Typos, einfache Features (Standard)
  • Merge Commit: Große Features, wo man später wissen will, dass das zusammenhängt
  • Rebase & Merge: Selten – nur wenn das Team das vereinbart

Automatisierung: Git Hooks vs GitHub Actions

Du möchtest, dass bestimmte Dinge automatisch passieren – z.B. Timestamps aktualisieren, Code formatieren, Tests laufen. Dafür gibt es zwei Systeme, die völlig unterschiedlich funktionieren:

🪝 Git Hooks

Was das ist: Scripts, die auf deinem PC automatisch laufen, wenn du bestimmte Git-Operationen machst (z.B. vor jedem Commit).

Wo sie leben: Lokal in .git/hooks/

Wann sie laufen: Nur wenn du lokal git commit machst

Problem: ❌ Läuft nicht in Remote-Sandboxen (Agent-Umgebung) ❌ Läuft nicht bei GitHub Web-Merges

⚡ GitHub Actions

Was das ist: Automatisierungs-Workflows, die auf GitHub's Servern laufen, wenn bestimmte Events passieren (z.B. Push zu main).

Wo sie leben: Im Repository in .github/workflows/

Wann sie laufen: Bei jedem Push, Merge, oder anderen GitHub-Events

Vorteil: ✅ Funktioniert überall ✅ Funktioniert mit Agenten ✅ Funktioniert mit GitHub Web

Vergleich: Wann nutze ich was?

Aufgabe Git Hook? GitHub Action?
Code vor Commit formatieren ✅ Ja (lokal) ❌ Nicht nötig
Timestamp aktualisieren nach Merge ❌ Funktioniert nicht in Sandbox ✅ Ja (perfekt)
Tests laufen lassen bei jedem Push ⚠️ Nur lokal ✅ Ja (alle Pushes)
Deploy auslösen bei Merge zu main ❌ Nicht möglich ✅ Ja (genau dafür)
Mit KI-Agenten arbeiten ❌ Funktioniert nicht ✅ Ja (immer)
Die Faustregel:

Für Teams mit Agenten: Nutze GitHub Actions, nicht Git Hooks. Actions funktionieren überall – lokal, in Remote-Sandboxen, auf GitHub.com, überall.

Praktisches Beispiel: Timestamp-Automation

🔧 In diesem Projekt

Das Problem: Nach jedem Merge soll der Timestamp in data.json aktualisiert werden, damit die Website sagt „Letztes Update: vor 2 Sekunden" statt „vor 3 Tagen".

Mit Git Hook: ❌ Funktioniert nur, wenn du lokal commitest. Wenn der Agent pusht oder du auf GitHub.com merkst, passiert nichts.

Mit GitHub Action: ✅ Workflow in .github/workflows/update-timestamp.yml triggert bei jedem Push zu /docs/. Timestamp wird automatisch aktualisiert. Überall.

GitHub Web, Desktop & Mobile – wo klickst du?

Git hat mehrere Oberflächen. Jede ist für andere Aufgaben optimiert.

18

Welche Oberfläche für welche Aufgabe?

🌐 GitHub.com (Web)

Das ist: Die Browser-Seite github.com. Keine Installation nötig, überall erreichbar.

Perfekt für: PRs reviewen, mergen, Dateien schnell editieren, Branches verwalten.

🖥️ GitHub Desktop

Das ist: Eine Desktop-App – grafisches Interface für Git-Operationen, ohne Terminal.

Perfekt für: Lokale Arbeit, Branches anschauen, Diffs vergleichen, Commits erstellen.

📱 GitHub Mobile

Das ist: Die App fürs Handy – GitHub im Miniformat.

Perfekt für: PRs reviewen unterwegs, Kommentare, im Notfall schnell freigeben.

⌨️ Terminal

Das ist: PowerShell, CMD, oder Unix-Shell – für die, die alles kontrollieren wollen.

Perfekt für: Agenten (die arbeiten immer im Terminal), wenn dir die UIs nicht reichen.

Im IT-Systemhaus – realistisch:

90% der Arbeit passiert in GitHub.com (Web). Das ist deine Zentrale. GitHub Desktop brauchst du, wenn du lokal entwickeln willst (selten). Terminal brauchst du fast nie. Handy ist praktisch, wenn du unterwegs einen PR freigeben musst.

GitHub.com – deine zentrale Kommandozentrale

Was du auf GitHub.com machst:

  • Pull Requests ansehen – Code-Changes, Diskussionen, Reviews
  • PRs reviewen und freigeben – „Looks good to me" oder Feedback geben
  • PRs mergen – Merge-Strategie wählen (Merge Commit, Squash, Rebase)
  • Branches verwalten – erstellen, löschen, anschauen
  • Dateien direkt editieren – schnelle Fixes, ohne lokal zu arbeiten
  • Commits direkt erstellen – wenn du das Web-Interface benutzt, statt Terminal
  • History ansehen – wer hat was wann geändert

Der typische Workflow auf GitHub:

  1. 1
    PR-Seite öffnen – Auf deinem Repo → „Pull Requests" Tab
  2. 2
    PR lesen – Title, Description, was wurde geändert
  3. 3
    „Files changed" anschauen – Zeige dir die Diffs (grün = neu, rot = entfernt)
  4. 4
    Kommentieren oder freigeben – Feedback geben oder „Approve"
  5. 5
    Merge-Strategie wählen und mergen – Grüner Button „Merge pull request" → Dropdown → Wahl → „Confirm merge"
GitHub Web-Tricks:
  • Drücke p auf einer PR-Seite, um zum PR zu gehen (Keyboard-Shortcut)
  • Mit ? siehst du alle Keyboard-Shortcuts
  • Im Diff-View: Rechts oben → „Split" statt „Unified" für zwei-spaltig Ansicht

Deep Dive: PRs reviewen und mergen

Was du in einem PR checkst:

  • Die Commits – Sind die aussagekräftig beschrieben? („Add" = neu, „Fix" = Bugfix, „Update" = Verbesserung)
  • Die Changes – Macht das inhaltlich Sinn? Sind da Fehler?
  • Die Tests – Wenn es Tests gibt, grün sie?
  • Die Dokumentation – Ist das, was der Agent gemacht hat, dokumentiert (in Code-Comments oder in der PR-Description)?

Merge-Strategie wählen (auf GitHub.com):

Wenn alles gut aussieht:

  1. Button „Merge pull request" (grün, oben rechts)
  2. Dropdown-Arrow → wähle eine Strategie
  3. „Confirm merge"
Was danach passiert:

Der Branch wird mit main zusammengefügt. Der Feature-Branch ist dann „merged" und wird nicht mehr gebraucht. GitHub bietet an, den Branch zu löschen – mach das immer.

Notfall: Ein Hotfix in Produktion

Der Kunde ruft an: „Die Live-Website ist kaputt!" Hier ist, was du tust:

🚨

Schritt 1: Fehler verstehen

Was ist kaputt? Ein Fehler-Screenshot? Einen Link zur Live-Website? Das brauchst du, um dem Agenten zu helfen.

🤖

Schritt 2: Agenten instruieren

„Hotfix nötig: Die CSS ist kaputt, weil ... Erstelle einen Branch `hotfix/css-bug`, behebe das, und öffne einen PR nach main."

⚡

Schritt 3: Schnell reviewen

Der Agent öffnet den PR. Du schaust schnell drauf – macht das Sinn? Ist es ein richtiger Fix oder nur ein Pflaster? Review schnell.

✅

Schritt 4: Schnell mergen

Merge sofort (normalerweise mit Squash & Merge). Der Deploy läuft automatisch. Live ist wieder gut.

Wichtig bei Hotfixes:

Das ist kein Test mehr – du reviewst schnell und gründlich genug, um sicher zu sein, dass es hilft und nicht mehr kaputtmacht. Hotfixes sind immer ein bisschen stressig, aber das ist normal.

Wie Git intern funktioniert

Git ist im Kern eine elegante Objekt-Datenbank. Jeder Commit, jede Datei, jedes Verzeichnis ist ein Objekt mit einem eindeutigen Fingerabdruck (SHA-1-Hash).

Die 4 Objekttypen:

Blob

Der Dateiinhalt – nur der Text, keine Metadaten. Wenn dieselbe Datei zweimal vorhanden ist, speichert Git sie nur einmal.

Tree

Ein Verzeichnis – zeigt, welche Blobs und Trees darin sind. Entspricht einem Ordner.

Commit

Ein Snapshot – zeigt auf einen Tree (den Zustand), den Parent-Commit, Autor, Zeit und eine Beschreibung.

Tag

Ein benannter Zeiger auf einen Commit – z.B. v1.0.0 für eine Release-Version.

Der Commit-Graph (DAG)

Commits bilden einen gerichteten azyklischen Graphen. Jeder Commit zeigt auf seinen Vorgänger. Branches sind dabei nur leichtgewichtige Zeiger.

Das wichtigste zum Mitnehmen:

Da Commits unveränderlich sind und in einer Kette stehen, kann Git alles rekonstruieren. Kein Commit ist verloren – selbst wenn du einen Branch löschst, sind die Commits noch im Reflog 30 Tage erreichbar.

Weitere Konzepte – für Fortgeschrittene

Staging Area

Eine Zwischenebene zwischen deinen Dateien und dem nächsten Commit. Du wählst bewusst, welche Änderungen reingehören – nicht alle.

HEAD

Ein Zeiger, der sagt: „Du bist hier." Normalerweise zeigt HEAD auf einen Branch, der auf einen Commit zeigt.

Fork

Eine Kopie eines fremden Repositories auf GitHub – auf Server-Ebene. Typisch bei Open-Source-Beiträgen.

Revert

Erzeugt einen neuen Commit, der einen früheren Commit exakt rückgängig macht. Die sichere Undo-Methode.

Praxis: Kunde Bauer GmbH – Monitoring-Tool

Szenario: Du sollst für Kunde Bauer GmbH ein PowerShell-Skript schreiben, das täglich Festplattenplatz prüft und den Admin alarmiert, wenn unter 10% freier Platz.

📁

Schritt 1: Projekt-Setup

Was passiert: Du erstellst ein neues Repository auf GitHub (kunde-bauer-diskcheck), clonest es lokal (oder der Agent macht das), und legst eine .gitignore an (damit Logs nie ins Repo kommen).

Nach Schritt 1: Ein leeres Repository, bereit zum Arbeiten.

✏️

Schritt 2: Erstes Skript

Was passiert: Du (oder der Agent) schreibt die Grundstruktur des PowerShell-Skripts. Es läuft lokal, macht einfache Disk-Checks.

Commit: „Add initial disk check script with basic logging"

Nach Schritt 2: Das Skript läuft, ist im Repo dokumentiert, andere können es sehen.

🔧

Schritt 3: Feature – E-Mail-Alerts

Was passiert: Der Agent erstellt einen Branch (feature/email-alerts) und implementiert E-Mail-Benachrichtigungen, wenn der Platz unter 10% fällt.

Commits:
• „Add email notification function"
• „Test email sending in lab environment"

Nach Schritt 3: Ein PR auf GitHub, warte auf dein Review.

👀

Schritt 4: Review & Merge

Was du tust: Du schaust den PR an, die Commits sind saubbar, die E-Mail-Funktion sieht gut aus. Du klickst „Approve" und „Squash and Merge".

Nach Schritt 4: Die E-Mail-Funktion ist in main, automatisch deployed.

🚨

Schritt 5: Hotfix – Crash beim Kunden

Was passiert: Zwei Wochen später ruft der Kunde an: das Skript crasht, wenn ein Netzwerk-Laufwerk nicht erreichbar ist. Hotfix nötig.

Der Agent: Branch hotfix/network-drive-check, implementiert einen Null-Check, testet lokal.

Du: Schaust schnell drauf, genehmigst, mergst sofort.

Nach Schritt 5: Der Kunde hat den Fix, der Live-Server läuft wieder stabil.

📚

Schritt 6: Ein neuer Kollege kommt

Was passiert: Nach 6 Monaten tritt ein neuer Kollege ein. Er braucht zu verstehen, was das Projekt ist. Er schaut sich nicht auf Word-Dokumente an – er schaut sich die Git-History an:

a3f9c12 Fix null reference when network drive is unreachable
8b2e4a1 Add email alert when disk space below 10%
3c7d9f0 Add initial disk check script with basic logging

Nach Schritt 6: Der Kollege versteht die komplette Geschichte des Projekts in 30 Sekunden.

Automatisierte Deployments mit Azure Key Vault

Ziel: Wenn ein PR auf main gemergt wird, sollen automatisch Dateien zu SharePoint (oder anderen Zielen) deployed werden — sicher, ohne hardcodierte Credentials.

Warum nicht einfach GitHub Secrets?

Das Problem: GitHub Secrets sind verschlüsselt, aber zentral im GitHub-Konto. Wenn mehrere Teams/Projekte arbeiten, ist die Credential-Verwaltung schnell chaotisch. Außerdem: keine echte Rotation, keine granularen Berechtigungen.

Die Lösung: Azure Key Vault — ein sicherer Tresor, den nur autorisierte Services nutzen können. Credentials werden dort verschlüsselt verwahrt und GitHub Actions holt sie nur zur Runtime.

Setup: Schritt für Schritt

Schritt 1: Azure Service Principal erstellen

Ein Service Principal ist eine Art "Roboter-Benutzer", der sich bei Azure anmelden kann ohne interaktive Login-Prompts.

az ad sp create-for-rbac \
  --name "testlearn-deployer" \
  --role "Contributor" \
  --scopes /subscriptions/{subscription-id} \
  --output json

Das gibt dir:

{
  "clientId": "xxxxxxxx-xxxx-...",
  "clientSecret": "***",
  "tenantId": "yyyyyyyy-yyyy-...",
  "subscriptionId": "zzzz..."
}

Schritt 2: Azure Key Vault anlegen

az keyvault create \
  --name "testlearn-kv" \
  --resource-group "my-rg" \
  --location "East US"

Schritt 3: Secrets in den Vault legen

az keyvault secret set \
  --vault-name "testlearn-kv" \
  --name "SharePointClientSecret" \
  --value "your-actual-secret-here"

az keyvault secret set \
  --vault-name "testlearn-kv" \
  --name "SharePointClientId" \
  --value "your-client-id-here"

az keyvault secret set \
  --vault-name "testlearn-kv" \
  --name "SharePointTenant" \
  --value "yourtenant.onmicrosoft.com"

Schritt 4: Service Principal Zugriff auf Key Vault geben

az keyvault set-policy \
  --name "testlearn-kv" \
  --spn {clientId-des-service-principals} \
  --secret-permissions get list

Schritt 5: GitHub Actions Secrets für Azure-Login

Gehe zu GitHub Repo → Settings → Secrets and variables → Actions und erstelle:

AZURE_CLIENT_ID = (aus Schritt 1)
AZURE_CLIENT_SECRET = (aus Schritt 1)
AZURE_TENANT_ID = (aus Schritt 1)
AZURE_SUBSCRIPTION_ID = (aus Schritt 1)
AZURE_KEYVAULT_NAME = testlearn-kv

Wichtig: Nur diese 5 Secrets in GitHub. Die echten SharePoint-Credentials bleiben im Key Vault!

GitHub Actions Workflow

Neue Datei: .github/workflows/deploy.yml

name: Deploy to SharePoint

on:
  push:
    branches:
      - main

permissions:
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      # 1. Code checken
      - uses: actions/checkout@v4

      # 2. Bei Azure anmelden (benutzt nur die 5 GitHub Secrets)
      - name: Azure Login
        uses: Azure/login@v1
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      # 3. Secrets aus Key Vault holen
      - name: Get secrets from Azure Key Vault
        uses: Azure/get-keyvault-secrets@v1
        with:
          keyvault: ${{ secrets.AZURE_KEYVAULT_NAME }}
          secrets: 'SharePointClientSecret,SharePointClientId,SharePointTenant'
        id: kvSecrets

      # 4. Dateien filtern (Whitelist)
      - name: Filter files for deployment
        run: |
          mkdir -p deploy_staging
          # Nur docs/ deployen
          cp -r docs/* deploy_staging/ 2>/dev/null || true
          # Nur PowerShell-Skripte deployen
          cp scripts/Get-SystemInfo.ps1 deploy_staging/ 2>/dev/null || true
          echo "Files ready for deployment:"
          ls -la deploy_staging/

      # 5. Deploy zu SharePoint
      - name: Deploy to SharePoint
        env:
          SP_CLIENT_ID: ${{ steps.kvSecrets.outputs.SharePointClientId }}
          SP_CLIENT_SECRET: ${{ steps.kvSecrets.outputs.SharePointClientSecret }}
          SP_TENANT: ${{ steps.kvSecrets.outputs.SharePointTenant }}
        run: |
          # Oder Python-Script aufrufen:
          # python3 scripts/deploy_to_sharepoint.py
          echo "Deployment gestartet..."
          # (hier kommt dein Deploy-Script rein)

      # 6. Bei Fehler: Notification (optional)
      - name: Notify on failure
        if: failure()
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '⚠️ Deployment zu SharePoint fehlgeschlagen. Siehe [Actions Log](https://github.com/${{ github.repository }}/actions/runs/${{ github.run_id }})'
            })

Python-Script für SharePoint-Upload

Datei: scripts/deploy_to_sharepoint.py

#!/usr/bin/env python3
import os
import requests
import json

# Credentials aus Umgebungsvariablen (von GitHub Actions)
client_id = os.getenv("SP_CLIENT_ID")
client_secret = os.getenv("SP_CLIENT_SECRET")
tenant = os.getenv("SP_TENANT")

# 1. Token von Azure AD holen
def get_access_token():
    url = f"https://login.microsoftonline.com/{tenant}/oauth2/v2.0/token"
    payload = {
        "client_id": client_id,
        "client_secret": client_secret,
        "scope": "https://graph.microsoft.com/.default",
        "grant_type": "client_credentials"
    }
    response = requests.post(url, data=payload)
    if response.status_code != 200:
        raise Exception(f"Auth failed: {response.text}")
    return response.json()["access_token"]

# 2. Dateien zu SharePoint hochladen
def upload_files(access_token, site_id, drive_id):
    headers = {"Authorization": f"Bearer {access_token}"}

    for file in os.listdir("deploy_staging"):
        file_path = os.path.join("deploy_staging", file)

        # Nur reguläre Dateien, keine Verzeichnisse
        if not os.path.isfile(file_path):
            continue

        with open(file_path, "rb") as f:
            # SharePoint REST API Endpoint
            upload_url = (
                f"https://graph.microsoft.com/v1.0/sites/{site_id}/"
                f"drive/root:/{file}:/content"
            )
            response = requests.put(upload_url, headers=headers, data=f.read())

            if response.status_code in [200, 201]:
                print(f"✅ Uploaded: {file}")
            else:
                print(f"❌ Failed: {file} — {response.text}")

if __name__ == "__main__":
    try:
        print("🔐 Authentifizierung...")
        token = get_access_token()

        print("📤 Uploading to SharePoint...")
        # Hier: site_id und drive_id müssen du konfigurieren
        # oder dynamisch abrufen
        upload_files(token, "YOUR_SITE_ID", "YOUR_DRIVE_ID")

        print("✅ Deployment erfolgreich!")
    except Exception as e:
        print(f"❌ Deployment fehlgeschlagen: {e}")
        exit(1)

Whitelist-Konfiguration

Datei: deploy_config.yml (optional, für mehr Kontrolle)

deploy:
  # Was deployt werden SOLL
  include:
    - "docs/**"              # Alle HTML/CSS/JS
    - "scripts/Get-SystemInfo.ps1"
    - "README.md"

  # Was NICHT deployt werden darf (Sicherheit)
  exclude:
    - "**/.env"
    - "**/*.log"
    - "**/node_modules"
    - "**/.git"
    - "CLAUDE.md"
    - ".github/**"

Sicherheits-Checkliste

Bevor du live gehst:
  • ✅ Service Principal hat nur Read-Zugriff auf Key Vault (nicht Admin!)
  • ✅ GitHub-Secrets enthalten nur Azure-Credentials, nicht die eigentlichen SharePoint-Secrets
  • ✅ Workflow hat permissions: contents: read (nur Lesen, keine Schreibzugriffe)
  • ✅ Agents (wie Claude) haben keinen Schreibzugriff auf .github/workflows/
  • ✅ deploy_config.yml ist im Repo und versioniert (damit änderungen sichtbar sind)
  • ✅ Log-Maskierung: GitHub maskiert Secrets automatisch — aber teste es!
  • ✅ Branch-Protection: PRs müssen reviewed werden vor Merge

Troubleshooting

„Authentication failed"
→ Service Principal ist nicht korrekt konfiguriert. Prüfe: az ad sp list --filter "displayName eq 'testlearn-deployer'"

„Permission denied on Key Vault"
→ Key Vault Policy ist nicht gesetzt. Wiederhole Schritt 4.

„Secrets sind leer in Workflow"
→ Die Azure-Login ist vor dem get-keyvault-secrets fehlgeschlagen. Schau in die Logs.
